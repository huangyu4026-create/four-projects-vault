## 二、这个外部年谱能给工程什么

这个资料最有价值的不是“它说了什么结论”，而是它已经把全书拆成了一个外部相对年份模型。

它提供三种底座能力：

| 能力 | 具体价值 |
|---|---|
| 相对年份骨架 | 给全书提供“元年到十九年”的外部年序候选 |
| 关键事件清单 | 把全书重要事件列成 201 个正文可复核节点 |
| 争议暴露 | 同一人物年龄、同一事件年份常有括号异说，正好进入争议表 |

所以它不是我们的主时间轴，也不是主轴底座，而是一个外部旁路工作底座：

```text
外部年谱负责提出候选。
红楼梦人工智能咨询工程负责回原文裁判。
主轴只接受原文时间和工程推论。
```

## 三、时间轴推进的总结构

以后时间轴推进采用“主轴 + 旁路”结构：

```text
主轴第1层：原文时间层
主轴第2层：工程推论层
旁路第1层：外部公开资料复核队列
旁路第2层：争议复核表
```

关系表：

| 层 | 来源 | 工程用途 |
|---|---|---|
| 原文时间层 | 原文明确时间词、节令、时辰、次日、过几日 | 最高可信时间锚 |
| 工程推论层 | 聚点顺序、时间块、场面切换、事件承接 | 补足前后流动 |
| 外部公开资料复核队列 | 外部公开资料中的年份模型 | 提供旁路候选和补漏线索，不进主轴 |
| 争议复核表 | 外部说法和原文/工程不一致处 | 保存疑点，不抹平，不进主轴 |

核心原则：

```text
外部年谱进入工程以后，先成为候选，不直接成为结论。
外部时间节点永远不作为主轴节点。
```

## 四、外部节点怎样进入我们的时间轴

每个外部节点必须经过这条链：

```text
external_node_id
-> 外部事件摘要
-> 候选 time_axis / time_block / 聚点
-> 回原文窗口
-> 判定状态
-> 只写入外部复核结果或争议复核表
```

判定状态建议：

| 状态 | 含义 |
|---|---|
| `原文硬锚已证` | 原文明确支持该时间点 |
| `原文弱锚可用` | 原文有节令/次日/前后承接，但不是完整日期 |
| `工程推论可用` | 原文没明说，可由时间块和事件顺序推定 |
| `外部候选待核` | 外部年谱有说法，工程尚未找到足够原文证据 |
| `外部说法冲突` | 外部年谱与原文、现有时间轴或其他外部说法冲突 |
| `旁路备查` | 资料有参考价值，但不进入主时间轴 |

## 五、候选对齐的经验

本轮已经生成 `红楼梦年谱_wikipedia_与现有time_axis候选对齐.csv`，它证明这个方向可行，但也暴露了风险。

有效例子：

| 外部节点 | 候选对齐 |
|---|---|
| 元年，八月十五，士隐赠银 | 能对到现有 `time_axis` 中第1回士隐赠银节点 |
| 元年，八月十六，雨村进京应试 | 能对到现有 `time_axis` 中雨村进京应试节点 |
| 元年，甄士隐梦太虚/石头入红尘 | 能对到现有神话入世节点 |

风险例子：

| 外部节点 | 机器误撞原因 |
|---|---|
| 三年，正月十五，英莲被拐 | 因“正月十五”误撞到第18回元妃省亲 |

结论：

```text
候选对齐只能负责召回。
真正入轴必须回原文裁判。
```

## 六、正式推进流程

### 第一步：外部年谱节点固定

把 `红楼梦年谱_wikipedia_外部年份模型.csv` 作为第一版外部年份模型。

每条记录都保留：

```text
来源
前八十回/后四十回
外部年份
外部日期/节令
事件摘要
原始行
```

### 第二步：和现有时间底座粗对齐

粗对齐目标不是定论，而是找候选：

```text
external_node_id
-> time_axis 候选
-> time_block 候选
-> atom_coordinates 聚点候选
```

候选对齐可以用：

| 方法 | 作用 |
|---|---|
| 日期词匹配 | 八月十六、正月十五、五更、秋、腊月 |
| 事件词匹配 | 士隐赠银、雨村进京、黛玉入府、秦可卿死 |
| 人物词匹配 | 林黛玉、贾宝玉、英莲、王熙凤等 |
| 顺序约束 | 外部年份顺序不能随意倒置 |

### 第三步：回原文复核

每个候选节点必须回到原文窗口：

```text
候选 time_axis
-> chapter_no
-> time_block_id
-> old_segment_no / atom_code
-> 原文上下文
```

复核时只问三件事：

1. 原文有没有明确时间？
2. 如果没有，前后是否足够推出？
3. 外部年份说法是否只是研究者推定？

### 第四步：反向修正主轴候选

外部节点本身不写入 `timeline_nodes`。复核之后，只允许发生两种动作：

| 动作 | 说明 |
|---|---|
| 主轴补漏提示 | 外部资料提示某处可能有原文时间锚，回原文证实后，由原文证据生成主轴节点 |
| 争议复核记录 | 外部资料和原文/工程推论不一致，写入争议表 |

主轴节点的来源只能是：

```text
原文时间证据
工程推论证据
```

外部节点只能作为旁路字段被记录在复核表中：

| 字段 | 说明 |
|---|---|
| `external_node_id` | 外部旁路节点 |
| `external_claim` | 外部资料的说法 |
| `checked_atom_code` | 已复核的聚拢段 |
| `checked_old_segment_no` | 已复核的聚点 |
| `review_result` | 补漏成立 / 外部冲突 / 仅备查 |
| `main_axis_action` | 不动作 / 提示原文建点 / 写争议 |

### 第五步：每个聚点挂时间

最终必须落回聚点：

```text
timeline_node_id
-> time_block_id
-> scene_id
-> atom_code
-> old_segment_no
```

这样时间轴才会真正进入红楼梦人工智能咨询工程。

## 七、这个底座如何推动全书时间轴

外部年谱清洗后的 201 条正文节点可以变成全书时间轴施工的“复核队列”。

推进顺序建议：

| 阶段 | 工作目标 |
|---|---|
| 1 | 先核第1回到第5回，验证外部节点、原文节点、聚点节点是否能闭合 |
| 2 | 核前八十回的硬日期节点，如八月十六、正月十五、二月二十二、四月二十六等 |
| 3 | 核人物年龄和年份争议点，不急于写死 |
| 4 | 核后四十回，单列版本段，不和前八十回强行揉平 |
| 5 | 形成全书主轴 `timeline_nodes`、`timeline_node_atom_edges`，同时另列外部复核表和争议表 |

这套推进的好处：

```text
外部资料给我们一张大地图。
原文给我们裁判权。
红楼梦人工智能咨询工程给我们落点。
争议表给我们保留复杂性。
```

## 八、最终结论

外部公开资料应该作为“外部旁路年份模型”使用。

它的工作身份是：

```text
不是答案。
不是主库。
不是原文。
不是主轴节点来源。
而是全书时间轴施工的外部候选地图和复核队列。
```

正式工程应当这样吸收它：

```text
外部年谱节点化
-> 与现有 time_axis 粗对齐
-> 回原文复核
-> 原文证实的，由原文证据进入主轴
-> 原文未证实或冲突的，只进外部复核表/争议表
-> 主轴时间节点再投回每个聚点
```

这就能把外部公开资料真正变成时间轴工程的推进工具。

## 九、矩阵法三轮推进结果

本轮已经按“外部节点不进主轴”的原则跑完三轮旁证矩阵。

| 轮次 | 方法 | 结果 |
|---|---|---:|
| 第一轮 | 外部节点全量撞内部 `time_axis / segments / time_blocks / variable_points` | 39 条进入复验队列 |
| 第二轮 | 对第一轮模糊/未设节点，回到 `segments.summary / quote` 搜原文段落点 | 新增 7 条进入复验队列 |
| 第三轮 | 对高价值节点做前后内部节点夹逼，只收明确事件短语命中 | 新增 3 条进入复验队列 |

总账：

```text
外部正文节点：201
三轮后进入复验队列：49
三轮后旁路保留/暂不用：152
```

第三轮经验：

```text
普通搜索已经到边界。
继续扩搜会增加误撞。
第三轮只适合做高价值节点夹逼，不适合全量硬压。
```

最终使用口径：

```text
49 条不是主轴节点。
它们只是复验队列。
复验通过以后，也只能由内部原文段、聚点、时间块生成主轴点。
外部节点编号只作为旁证来源记录。
```
