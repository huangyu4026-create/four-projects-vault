## 一句话总原则

历史设计阶段曾经把红楼梦工程理解为双头部：

```text
聚拢查询头：用聚拢法组织现场。
坐标查询头：用坐标分析法计算落点、距离和共场。
```

这段只保留历史解释价值。当前活动线不再把它作为入口分流依据。

## 共同底座

两个头部共同遵守完整八步法：

```text
归一 -> 收点 -> 交集 -> 路由门 -> 分类 -> 回原文 -> 入材料池 -> 写答案
```

八步法是完整方法，覆盖读题、取词、归一、收点、交集、查库、回原文和入材料池。

## 取词前共同方法

取词之前，两头完全相同：

1. 读题。
2. 判断题型。
3. 拆对象。
4. 定主轴。
5. 人物归一。
6. 形成主查词、归一词、扩展词、背景词、排除词。
7. 形成强复合轴。
8. 判断是否需要子问题。
9. 判断是否需要两两比较。
10. 判断是否需要穷尽、距离、共场、第一次、最早、全书扫描。

这一段不分聚拢和坐标。差别从“取完词以后怎么用库”开始。

## 取词后总分流

```text
入口词包
  -> 聚拢查询头：候选段落 -> 聚拢单元 -> 聚拢事件 -> 聚拢现场 -> 材料池
  -> 坐标查询头：词位/变量/字位 -> atom/scene/event/time_block -> 距离/共场 -> 材料池
```

## 核心差异矩阵

| 差异项 | 聚拢法 | 坐标分析法 |
|---|---|---|
| 中心库 | 语义聚拢中心库 | 坐标查询中心库 |
| 主要对象 | 候选段落、聚拢单元、事件、现场 | atom、变量点、词位、字位、容器 |
| 核心动作 | 组织现场 | 收点、测距、共场 |
| 交集方式 | W32/W33 聚拢单元/事件交集 | atom/scene/event/time_block 交集 |
| 放大方式 | 聚拢现场放大、缩小、串域 | 容器归属放大 |
| 距离理解 | 场景关系强弱 | 原子段/场/事件/时间块距离 |
| 入池前角色 | 判主证、旁证、背景、误召回 | 判坐标候选、距离候选、共场候选 |
| 输出身份 | 聚拢材料 | 坐标材料包 |

## 聚拢法

聚拢法的核心不是计算距离，而是组织现场。

它的用库思路是：

```text
问题/方法
-> 候选段落
-> 语义聚拢中心库
-> W32 聚拢单元
-> W33 聚拢事件
-> 聚拢现场组织
-> 材料池四态裁判
-> 原文追证
```

聚拢法适合：

1. 判断人物、物象、空间、事件是不是在同一现场链里。
2. 把散落的候选材料组织成场景关系。
3. 区分主证、旁证、背景、误召回。
4. 处理叙事现场、事件现场、人物互动、材料角色。

聚拢法的原则：

1. 不只看词命中，要看候选是否回到聚拢单元和聚拢事件。
2. 不把同回共现当同场。
3. 聚拢库负责组织现场，不负责直接写答案。
4. 进入材料池以后，还要四态裁判和原文追证。

## 坐标分析法

坐标分析法的核心不是组织语义现场，而是计算落点、距离和共场。

它的用库思路是：

```text
入口词包
-> 词位穷尽防漏
-> 变量坐标定点
-> atom / scene / event / time_block 收点
-> 两两交集 / 近邻测距
-> 容器归属放大
-> 原文句回查
-> 材料池裁判
```

坐标分析法适合：

1. 查某个字词、短语、人物、物件、季节、空间在哪里。
2. 查两个对象离得最近的地方。
3. 查人物、物象、季节、空间是否同原子段、同场、同事件、同时间块。
4. 做全书穷尽扫描、按回序和字位排序。

坐标分析法的原则：

1. 词位负责防漏，坐标变量负责定点。
2. 变量未登记，不等于原文没有。
3. 精确变量不能替代全文词位和原文裁判。
4. 最近、靠近、同场，必须区分 atom、scene、event、time_block 的距离层级。
5. 坐标门只完成收点、补点、测距、共场和容器归属；候选仍要回材料池和原文裁判。

## 设计边界

不能把聚拢法改造成坐标法。

不能把坐标法改造成聚拢法。

聚拢法回答“这些材料如何组成现场”。

坐标分析法回答“这些对象落在哪里、距离多近、是否共场”。

两者最后都必须回到材料池和原文裁判。

## 后续设计原则

以后设计两个程序时，必须遵守：

1. 取词前共用同一套八步思考。
2. 取词后明确分头。
3. 聚拢头只增强聚拢法，不往坐标测距方向乱长。
4. 坐标头只增强坐标分析法，不往聚拢现场组织方向乱长。
5. 两边产物都回材料池，不直接写答案。
6. 所有结论都回原文裁判。

## 方法论：从真实流程看

聚拢法的方法论：

```text
候选不是答案；候选必须回到聚拢单元、聚拢事件和原文锚点，才能变成材料。
```

它解决的是：

1. 候选段落如何归并。
2. 哪些候选属于同一个现场。
3. 哪些材料是主证、旁证、背景或误召回。
4. 子问题如何各自过账。
5. 材料如何进入材料池四态裁判。

坐标分析法的方法论：

```text
命中不是答案；命中必须落到 atom、变量点、词位、字位、容器和距离层级，才能变成坐标材料。
```

它解决的是：

1. 对象在哪里。
2. 词有没有被穷尽查到。
3. 变量有没有坐标定点。
4. 两个对象有多近。
5. 多个对象是否同 atom、同 scene、同 event、同 time_block。

## 设计论：两个引擎应该怎样长

聚拢头应该继续长成：

```text
叙事现场组织引擎。
```

它的下一步增强，不应是重复坐标头的距离计算，而应是：

1. 提升 `segments` 摘要和原文锚点质量。
2. 提升 `segment_cluster_units` 的场景组织能力。
3. 提升 `event_segments` 和 `evidence_edges` 的事件解释能力。
4. 把候选段落更稳定地归并到 W32/W33。
5. 给材料角色更硬的判定字段：主证、旁证、背景、误召回、需补证。
6. 让每个子问题都有材料覆盖状态，不让总答案吞掉子问题。
7. 把“可摘抄材料”和“只是候选线索”分得更硬。

坐标头应该继续长成：

```text
全文结构测量引擎。
```

它的下一步增强，不应是重复聚拢头的现场解释，而应是：

1. 提升 `variable_dictionary` 的变量命名和类型质量。
2. 提升 `variable_points` 的覆盖和证据强度分层。
3. 提升 `v_term_hits` 和变量点之间的互证关系。
4. 提升 `v_char_hits` 回原文句子的稳定性。
5. 强化 `v_atom_distance_directed` 的距离解释。
6. 强化 same_atom、same_scene、same_event、same_time_block 的共场判定。
7. 生成更清楚的坐标材料包，让材料池知道这条材料是距离候选、共场候选还是词位候选。

## 实用论：什么问题先走谁

优先走聚拢头的问题：

1. 问“这个场景里发生了什么”。
2. 问“谁和谁在一个现场里有什么关系”。
3. 问“某个物象在事件里起什么作用”。
4. 问“哪些材料能组织成一个叙事链”。
5. 问“这条材料能不能当主证”。

优先走坐标头的问题：

1. 问“某个词在哪里”。
2. 问“某个人物在哪里出现”。
3. 问“两个对象最近在哪里”。
4. 问“几个人、物、季节、空间是否同场”。
5. 问“第一次、最早、全书哪里、高密度分布”。

需要双头协作的问题：

1. 坐标头先穷尽落点，聚拢头再组织现场。
2. 坐标头先算最近点，聚拢头再判断这个近是否有叙事意义。
3. 聚拢头先判断现场链，坐标头再验证对象距离和变量交集。

## 发展论：真实短板决定发展方向

聚拢头下一步最该发展：

1. 摘要规范：摘要必须更稳定地保留地点、人物、动作、原文锚点。
2. 聚拢单元质量：W32 要更像“现场单位”，减少宽泛场景名。
3. 聚拢事件质量：W33 要更清楚地表达事件阶段和材料角色。
4. 材料池前分层：候选、可摘抄、可主证、需补证要更硬。
5. 子问题覆盖：复杂题每个子问题都要有覆盖状态。

坐标头下一步最该发展：

1. 变量精确度：硬变量、提示变量、背景变量要分得更清。
2. 词位和变量互证：有词无变量、有变量无词都要标明。
3. 距离解释：距离不只给数字，还要解释 same_atom / same_scene / same_event / same_time_block。
4. 多轴矩阵：人、人、物、季节、空间、事件要能稳定两两成表。
5. 坐标材料包可读性：候选进入材料池前，必须说明为什么这条候选有用或需要降权。

## 中心攻坚任务

聚拢法的中心攻坚任务：

```text
把多段材料组织成叙事链，判断材料主次，形成整体现场解释。
```

这不是说坐标法不能解释原文。坐标法定位到 atom 和原文以后，也能判断材料角色。但聚拢法的核心任务，是把多个候选段、多个现场、多个事件阶段组织起来，回答“这一组材料怎样成为一个叙事现场”。

所以以后说“优化聚拢法”，真正要优化的是：

1. 多段材料怎样合并成一条现场链。
2. 多个候选怎样判主证、旁证、背景、误召回。
3. 一个物象、人物、空间、事件在现场中的角色怎样定。
4. 一个复杂问题怎样拆成几个现场，再合成总判断。
5. 聚拢单元和聚拢事件怎样更像真正的叙事结构。
6. 材料池前，怎样把“可写主证”和“只作背景”分得更准。

坐标法的中心攻坚任务：

```text
把对象更精确、更快速、更全量地落到坐标，并计算交集、距离、共场和分布。
```

这不是说坐标法不能解释叙事意义。坐标法可以在定位后回原文解释。但坐标法的核心任务，是先把全书材料找准、找全、排好、量出来。

所以以后说“优化坐标法”，真正要优化的是：

1. 词位穷尽更全，避免漏字、漏词、漏短语。
2. 变量坐标更准，人物、物象、空间、季节、动作都能稳定落点。
3. 距离层级更清，same_atom、same_scene、same_event、same_time_block 不混。
4. 多对象交集更快，人-人、人-物、人-季节、人-空间能直接成矩阵。
5. 首次、末次、最近、高密度分布能稳定输出。
6. 坐标材料包能清楚说明这条材料为什么被选中、为什么降权、为什么需要回原文。

两个攻坚任务的边界：

```text
聚拢法攻坚“材料怎样组织成现场链”。
坐标法攻坚“对象怎样精确落点并被测量”。
```

如果一个优化项让聚拢法变成单纯测距工具，它走偏了。

如果一个优化项让坐标法变成宽泛叙事解释器，却没有提升落点、速度、全量、距离和交集，它也走偏了。

## 坐标完整轮与聚拢二轮复核的真实边界

这里必须区分两个东西：

```text
坐标材料门：只完成坐标取材、入池凭证，停在材料池入口。
完整坐标查询头：坐标材料门之后，继续接同一个材料池四态、精读材料词、写作前原文追证和红楼解语。
```

所以不能把“坐标材料门的截断状态”误认为“坐标查询头的全部能力”。

工程里的 `000C_进入坐标查询_新窗口强制入口_候选正式头.md` 已经写清：

```text
新头：进入坐标查询
  -> 坐标分流令牌
  -> 查询词路
  -> 人物归一强令牌
  -> 坐标材料门
  -> 全文词位库 / 变量投影库 / 字位桥表
  -> 距离 / 共场 / 交集
  -> 原子段与原文句材料候选
  -> 同一个材料池入口
  -> 原文裁判
  -> 精读材料词
  -> 写作前原文追证
  -> 红楼解语
```

这说明完整坐标查询不是一个短定位工具。它和聚拢头一样，也有后面的材料池、原文裁判、精读和最终写作能力。

### A轮坐标完整流程能不能自己完成

能。

如果 A轮走的是完整坐标查询头，它不仅能判断：

1. 是否同 atom。
2. 是否同 scene。
3. 是否同 event。
4. 是否同 time_block。
5. 距离多近。
6. 原文锚点在哪里。

它还可以继续进入同一个后身：

```text
材料池四态 -> 精读材料词 -> 写作前原文追证 -> 红楼解语
```

因此，A轮坐标完整流程本身可以完成材料取舍、原文追证和最终回答。不是必须交给 B轮聚拢法才能判断材料价值。

### 那么B轮聚拢法还有什么意义

B轮聚拢法的意义不是“替 A轮做后半段”，也不是“复核 A轮已经能完成的材料池裁判”。

B轮只在三种情况下有意义：

1. 二次研究：A轮已经给出坐标答案，用户还想从聚拢法角度看这组材料怎样组成更大的现场链。
2. 争议复核：A轮结果有疑义，需要另一套库法从 W32/W33 聚拢单元、聚拢事件、现场链角度做独立验证。
3. 跨方法互证：一个重要结论要同时具备“坐标可测量性”和“聚拢现场解释力”，才进入更高等级研究结论。

这时 B轮接收的不是“尚未完成的半成品”，而是“完整 A轮结论的二次审读对象”。

### B轮追问的转译原则

A轮给 B轮生成追问时，不能把 A轮自己的优势问题原样交出去。

这些问题属于坐标法优势区，不应该作为 B轮主问题：

1. 哪些候选最高密度。
2. 哪些对象距离最近。
3. 哪些对象同 atom / scene / event / time_block。
4. 哪些回目分布最集中。
5. 哪些词位、变量点、字位命中最多。

这些已经是 A轮应该完成的事。

B轮要接的，不是“再查高密度候选”，而是把 A轮结论翻译成聚拢法能回答的问题：

```text
A轮问题：哪里高密度？
B轮问题：这个高密度区域是否构成一个叙事现场链？

A轮问题：哪里最近？
B轮问题：这个最近点在叙事中有没有材料意义，还是只是坐标近？

A轮问题：是否同场？
B轮问题：这个同场在聚拢事件里是主场、边场、背景，还是误召回？

A轮问题：哪些点最多？
B轮问题：这些点能不能组织成一个可解释的现场结构？
```

所以 A轮给 B轮出题时，必须避开“计算题”，改成“现场意义题、材料角色题、反证题、叙事链题”。

### B轮追问生成方法

A轮结束后，生成 B轮问题时按四步走：

1. 先摘出 A轮核心结论：A轮到底说什么成立。
2. 再摘出 A轮证据形态：它靠的是同 atom、同 scene、同 event、最近距离、高密度，还是全量分布。
3. 再找 A轮最脆弱的地方：是原文句不够强、只是背景共现、变量提示太弱、同事件太宽，还是材料角色不明。
4. 最后把问题转译给 B轮：不要再问计算是否成立，只问现场意义、材料主次、反证风险、叙事链是否成立。

公式是：

```text
A轮结论 + A轮证据形态 + A轮不放心点
-> B轮反证 / 二次研究 / 跨方法互证问题
```

### B轮追问例子库

例一：同场题。

A轮原问题：

```text
宝玉、黛玉、手帕、眼泪是否同场？
```

A轮应该完成：

```text
查词位、变量点、atom / scene / event / time_block，判断是否同场、距离多近、在哪些原子段。
```

B轮不能问：

```text
请再查宝玉、黛玉、手帕、眼泪是否同场。
```

B轮应该问：

```text
进入聚拢查询：
A轮已经判定宝玉、黛玉、手帕、眼泪存在若干同场坐标点。
请不要重新计算同场。
请只判断这些同场点在聚拢现场中是什么材料角色：
它们是宝黛关系的主证，还是只是物象、眼泪、人物同时出现的背景材料？
有没有前后文说明手帕或眼泪并不属于宝黛关系核心现场？
哪些材料应降级为旁证、背景或误召回？
```

例二：最近题。

A轮原问题：

```text
林黛玉和黄字离得最近的是哪一回？
```

A轮应该完成：

```text
人物归一、黄/黃/颜色字词位穷尽、坐标距离计算，给出最近 atom、回目、原文句。
```

B轮不能问：

```text
请再查林黛玉和黄字最近是哪一回。
```

B轮应该问：

```text
进入聚拢查询：
A轮已经判定林黛玉与黄字的最近坐标点。
请不要重新测距离。
请只判断这个最近点在聚拢现场里有没有解释价值：
它是否构成黛玉与黄色意象的有效材料？
还是只是字面颜色、物件细节、同段偶然共现？
如果不能作主证，请说明应降级为背景、旁证还是误召回。
```

例三：高密度题。

A轮原问题：

```text
宝玉、黛玉、秋、花最密集地聚在哪里？
```

A轮应该完成：

```text
全量扫描，计算高密度回目、原子段、场、事件和分布。
```

B轮不能问：

```text
请围绕高密度候选继续查哪里最密。
```

B轮应该问：

```text
进入聚拢查询：
A轮已经给出宝玉、黛玉、秋、花的高密度区域。
请不要重新计算密度。
请只判断这些高密度区域是否构成“秋花宝黛”的叙事现场链：
它们是同一条情感/事件现场，还是若干坐标散点？
秋和花在这些材料中是核心意象，还是季节背景和普通物象？
有没有理由说明“秋花宝黛”这个组合需要降级或拆开处理？
```

例四：存在题。

A轮原问题：

```text
黛玉是否亲口说过自己做了一个梦？
```

A轮应该完成：

```text
查梦、做梦、自述、说话边界、人物归一、原文句，判断严格存在或不成立。
```

B轮不能问：

```text
请再查黛玉有没有说自己做梦。
```

B轮应该问：

```text
进入聚拢查询：
A轮已经判断“黛玉亲口自述梦”严格证据不足。
请不要重新做全文是否存在计算。
请只围绕A轮排除或降级的梦境材料，寻找最可能推翻A轮的边界：
哪些材料容易被误判为亲口自述？
哪些是梦中说话、叙述者说明、醒后反应、他人转述？
有没有一条材料能使A轮从“不成立”改为“边界成立”或“需补证”？
```

例五：物象共现题。

A轮原问题：

```text
鸡和花有没有同场材料？
```

A轮应该完成：

```text
鸡/雞/鷄与花的词位穷尽、变量点、同 atom / scene / event 计算。
```

B轮不能问：

```text
请再查鸡和花是否同场。
```

B轮应该问：

```text
进入聚拢查询：
A轮已经找出鸡与花的同场坐标材料。
请不要重新计算同场。
请只判断这些同场在聚拢现场中的意义：
它们是生活物象偶然共现，还是进入人物关系、场面气氛或事件转折？
有没有材料虽然坐标同场，但在叙事上完全不能形成“鸡与花”的关系？
```

例六：阶段比较题。

A轮原问题：

```text
宝黛关系中，春天和秋天哪个更接近花？
```

A轮应该完成：

```text
按春/秋、宝玉/黛玉、花做全量坐标分布、距离和共场统计。
```

B轮不能问：

```text
请再比较春天和秋天哪个更接近花。
```

B轮应该问：

```text
进入聚拢查询：
A轮已经给出春/秋、宝黛、花的坐标分布差异。
请不要重新统计距离和密度。
请只判断这种差异在聚拢现场里是否有叙事意义：
春花材料和秋花材料分别承担什么现场角色？
哪个更能进入宝黛关系主线？
有没有坐标上接近但只是季节背景的材料需要降级？
```

### A轮生成B轮反证委托题

更准确的双轮协作，不是让 B轮重复 A轮的问题，而是让 A轮在完成自己的完整判断后，自动生成一个给 B轮的“反证委托题”。

也就是说：

```text
A轮负责提出结论。
A轮同时提出自己最不放心的地方。
B轮不重做原题，只攻击 A轮结论最脆弱的地方。
```

如果 A轮判断“存在”，B轮的问题不是“是否存在”，而是：

```text
有没有理由说明这个存在只是坐标同容器，而不是原文现场成立？
有没有反例说明它只是背景、擦边、同回、同事件边缘材料？
有没有前后文证明这个材料不能作主证？
```

如果 A轮判断“最近”，B轮的问题不是“谁最近”，而是：

```text
这个最近是否有叙事意义？
是不是只是字位/原子段距离近，但现场关系并不强？
有没有另一个聚拢现场比这个坐标最近点更适合作为解释材料？
```

如果 A轮判断“可答”，B轮的问题不是“再答一遍”，而是：

```text
这个答案还有没有未覆盖的子问题？
主证是否被上下文削弱？
有没有被 A轮降权但聚拢现场意义更强的材料？
有没有反证、旁证或背景材料会改变答案等级？
```

所以 B轮最好的身份不是“复核员”，而是“反证员 / 压力测试员 / 二次审读员”。

### 反证委托题格式

A轮交给 B轮的重点，不是原题，而是这个包：

```json
{
  "handoff_name": "A轮结论反证委托包",
  "original_question": "原问题",
  "a_round_core_judgment": "A轮完整坐标查询形成的核心结论",
  "a_round_confidence": "强 / 中 / 弱 / 需补证",
  "a_round_main_evidence": [],
  "a_round_weak_points": [
    "A轮自己认为最可能出问题的地方"
  ],
  "b_round_counter_questions": [
    {
      "counter_question_id": "BQ001",
      "counter_question": "请寻找这个结论不成立、只能降级、或需要补证的理由。",
      "attack_target": "主证 / 同场判断 / 最近判断 / 材料角色 / 原文上下文",
      "allowed_scope": "只围绕A轮主证、弱点、相关聚拢单元和事件链，不重跑原题。",
      "expected_output": "支持A轮 / 降级A轮 / 推翻A轮 / 需补证"
    }
  ]
}
```

这个格式能避免 B轮变成重复劳动。

它让 B轮只做一件事：

```text
从聚拢法角度寻找 A轮结论的漏洞、降级理由、反证材料和更强解释链。
```

### A轮交给B轮的正确内容

A轮如果要交给 B轮，交的应该是完整坐标结论，而不是让 B轮替它完成流程。

交接包应包含：

1. A轮最终判断：可答 / 需补证 / 不足以答。
2. A轮材料池四态结果。
3. A轮精读材料词。
4. A轮写作前原文追证摘抄。
5. A轮最终候选或答案草案。
6. 关键坐标证据：`atom_code / segment_no / scene_id / event_id / time_block_id / distance_basis / pairwise_matrix`。
7. A轮认为最强的主证、旁证、背景、反证或误召回。
8. 需要 B轮特别看的争议点。

### 交接包的建议格式

```json
{
  "handoff_name": "A轮完整坐标结论二次审读包",
  "question": "原问题",
  "from_head": "coordinate",
  "to_head": "aggregation_second_read",
  "handoff_rule": "B轮不替A轮完成流程，只对A轮完整结论做二次聚拢审读。",
  "a_round_final_status": "可答 / 需补证 / 不足以答",
  "a_round_answer_or_core_judgment": "A轮坐标查询已经形成的核心判断",
  "a_round_material_pool_result": [],
  "a_round_close_reading_terms": [],
  "a_round_original_reread_passages": [],
  "coordinate_evidence": [],
  "dispute_points_for_b_round": [
    "是否存在更大的聚拢现场链",
    "A轮主证是否在聚拢事件里仍然居中",
    "是否有坐标强但叙事边缘的材料需要降级",
    "是否有坐标弱但聚拢现场意义强的材料需要补看"
  ],
  "hard_stop": "B轮是二次审读，不覆盖A轮；如结论不同，形成双头差异报告。"
}
```

### B轮输出应该是什么

B轮输出不是替代答案，而是二次审读结果：

1. `B1_聚拢二次审读意见.csv`：是否支持 A轮结论。
2. `B2_双头差异表.csv`：坐标强而聚拢弱、坐标弱而聚拢强、两头一致、两头冲突。
3. `B3_聚拢现场链补充.md`：如果 A轮答案需要更大的现场解释，这里补充。
4. `B4_最终采用建议.md`：采用 A轮、采用 B轮、合并双头，还是标为需补证。

### 修正后的价值判断

完整坐标头本身可以走完全流程。B轮不是必需后身。

双轮机制真正的价值是：

```text
坐标轮负责生成一套完整的、可测量的研究结论。
聚拢轮负责在需要时生成另一套现场组织角度的二次审读。
两轮不是前后半身，而是两套方法的互证。
```

如果只是普通坐标题，A轮完整坐标查询自己完成即可。

如果是重大题、争议题、解释型题，才启动 B轮聚拢二次审读。

## 服务对象论

聚拢头服务：

```text
解释型、现场型、叙事型、材料裁判型问题。
```

它最终服务的是“如何理解这组材料”。

坐标头服务：

```text
定位型、测量型、交集型、穷尽型问题。
```

它最终服务的是“这些对象在全书哪里相遇、相距多近、分布如何”。

## 科学家的设计判断

一个问题进入工程以后，不应该先问“哪个更高级”。

应该先问：

```text
我要的是现场理解，还是结构测量？
```

如果要现场理解，用聚拢头做主引擎。

如果要结构测量，用坐标头做主引擎。

如果要又准又能解释，先坐标后聚拢，或先聚拢后坐标，最后都回材料池和原文。

## 最终成熟状态

成熟的聚拢头：

```text
能把多段复杂候选组织成叙事现场链，判断主证、旁证、背景、误召回，并形成整体现场解释。
```

成熟的坐标头：

```text
能把全书对象精确、快速、全量地落到 atom / scene / event / time_block，计算距离、共场、交集和分布，并生成坐标材料包。
```

成熟的双头部：

```text
坐标头负责把对象找准、量清、排好。
聚拢头负责把材料组织成现场链、讲清主次。
材料池负责把材料分级。
原文负责把结论定死。
```

## 最高设计问题

以后任何新功能进入双头部之前，必须回答：

1. 它是增强聚拢法的叙事现场链组织，还是增强坐标法的精确落点、测距、共场和分布？
2. 它属于取词前共同方法，还是取词后分头方法？
3. 它用的是哪个中心库或哪张表？
4. 它输出的是聚拢材料、坐标材料包，还是材料池裁判结果？
5. 它有没有回原文？

答不清楚，就不能进入主工程。
