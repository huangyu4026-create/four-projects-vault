## 材料呈现方式层

材料呈现方式层是坐标工程的中间判别层。它不另开入口，仍只读坐标查询唯一中心大库；它的作用是先判断一个聚拢段“这段文字是怎样把材料呈现出来的”，再挂接人物、物象、空间、事件、话语、描写内部子类等查询编码。

顶层呈现方式只放互不打架的大类：

| 呈现方式 | 查询编码（variable_type） | 功能 |
|---|---|---|
| 话语对话呈现 | `presentation_dialogue` | 凡是开口、对话、转述、谁说给谁听、话里说了什么，先转模板 11。 |
| 描写呈现 | `presentation_description` | 容貌、服饰、景物、室内、室外、环境氛围、心理、动作、神情声态、器物细描、颜色感官。心理、空间环境、物象细节都属于描写内部子类，不和描写并列。 |
| 事件叙述呈现 | `presentation_event_narration` | 查发生了什么、事情怎样起、怎样转、怎样收束；需要再交叉 `event / chapter_no / global_atom_order`。 |
| 诗文判词谶语呈现 | `presentation_poem_prophecy` | 综合概念是“成文/预示性文本”：诗、词、曲、令、联、偈、题咏、判词、梦兆、谶语、签诗、花名签、灯谜、册判、红楼梦曲合并查。底层再交叉 `poem / prophecy`。 |

挂接轴不要和顶层呈现方式混淆：

```text
人物挂接轴：person_identity / scene_person_hint
物象挂接轴：object_fullscan_item / object_fullscan_class / object_fullscan_cooccurrence_candidate / scene_object_hint
茶酒药食方谱轴：recipe_formula / recipe_formula_class / recipe_formula_effect / recipe_spectrum
茶酒药食实物轴：drug_food_fullscan_material / drug_food_fullscan_class / drug_food_formula_candidate
空间挂接轴：space / scene_space_hint
事件挂接轴：event
描写内部子类轴：description_*
话语内部轴：speech_*
功能评价挂接轴：relationship_type / emotion_symbolic_function / function / note_type / note_dimension / perspective
```

查询次序：先用 `presentation_*` 判断材料呈现方式；再用人物、物象、空间、事件、话语、描写子类、功能评价轴做两两交叉；最后回 `atom_code / scene_id / global_atom_order` 和原文裁判。

按查证证据结构排题：题目进来后，不先散查词，先判断证据应该落在哪条结构上。坐标工程的查证证据结构先立四条主梁：人物结构、描写结构、推进结构、证据结构。结构不是另开库；结构是让机器知道先用哪组查询编码、怎样交叉、最后回哪一层原文坐标裁判。

| 查证证据结构 | 主要解决什么问题 | 包含哪些子入口 | 主查询编码 | 取证办法 |
|---|---|---|---|---|
| 人物结构 | 查某人、谁和谁、人物关系、人物交集、人物与物/事/地/话的关系 | 人物单体、人物关系、人物交集、人物话语、人物与物象、人物与空间、人物与事件 | `person_identity`；关系题交叉 `relationship_type`；开口题交叉 `speech_*` | 先做人名归一，再查人物编码；用 `old_segment_no / atom_code / scene_id / event_id / global_atom_order` 判断同聚点、同聚拢段、同场定位、同聚拢事件或近距；最后回原文裁判关系是否成立 |
| 描写结构 | 查文本怎么写人、写物、写景、写空间、写心理、写动作、写神情声态 | 容貌、服饰、景物、室内、室外、环境氛围、心理想法、心理情绪、动作身段、神情声态、器物细描、颜色感官 | `presentation_description` + `description_*` | 先确定是描写呈现，再用 `description_*` 和 `person_identity / object_fullscan_* / space` 交叉；同聚点优先，不足再放宽到聚拢段和同场定位；最后回描写原文看描写对象是否确实成立 |
| 推进结构 | 查一条线如何发展、关系怎样变化、事件怎样推进、命运如何前后连接 | 时间推进、事件推进、关系推进、空间移动、话语推进、诗文判词谶语预示推进、功能评价推进 | `chapter_no / global_atom_order` + `event / relationship_type / function_commentary_axis`；涉及开口交叉 `speech_*` | 先把线索拆成若干聚点，再按全书顺序排列；用聚点、聚拢段、聚拢事件、聚拢场逐层看前后连接；每一个推进点都要回原文，不能只凭排序自动下结论 |
| 证据结构 | 根据一个问题，如何回到原文里把证据找出来 | 问题拆解、查询编码定位、聚点回跳、原文窗口、上下文裁判、证据摘录、材料入池 | `variable_type / matched_text / old_segment_no / atom_code / scene_id / event_id / chapter_no / global_atom_order` | 先把问题翻成查询编码，再用编码交叉找到聚点；从聚点回到原文窗口，摘出原句和上下文；最后判断这段原文能不能证明问题，而不是只看编码命中 |

四结构挂接规则：话语对话、诗文判词谶语、功能评价都不能漂在外面。问谁说、谁听、谁和谁说，先挂人物结构，再用 `speech_*` 精查；问一句话、一首诗、一段判词如何预示后文，挂推进结构；问某段为什么重要，挂功能评价轴，但必须落回人物、描写、推进或证据结构中的一个；问证据怎么找、原文在哪里、该摘哪一段、能否证明问题，直接进入证据结构。

模板归位口诀：问“谁/谁和谁/谁和什么有关”，先进人物结构；问“怎么写、怎样描写、哪里写衣服容貌心理动作景物”，先进描写结构；问“如何发展、前后顺序、关系变化、事件线、命运线”，先进推进结构；问“证据怎么找、原文在哪里、该摘哪一段、能不能证明问题”，进入证据结构。复杂题先拆成几个小问题，每个小问题都要落到四结构之一，再进具体模板。

推进结构已经形成正式用法：凡问先后、阶段、长线、命运线、关系变化、事件发展、某人某物在全书中怎样前后出现，都先进推进结构。推进结构不是另开入口，仍只读坐标查询唯一中心大库；它的核心动作是把问题拆成若干聚点，用 `global_atom_order`、时间主轴、故事进程轴和距离函数排序，再与人物、话语、描写、事件等编码交叉，最后回原文裁判。

故事进程轴已经入库并完成编码定位表打点：它不是三个库，而是坐标查询唯一中心大库里的编码投影 + 三张结构表。机器入口先看 `variable_points.variable_type` 的五个查询编码：`story_progression_scene_group / story_progression_event / story_progression_unit / story_progression_segment / story_progression_point`；查到编码点后，再接 `story_progression_nodes / story_progression_edges / story_progression_layer_stats` 看层级、摘要和前后关系。这个轴不用外部资料，只用工程内部聚拢层级，把全书拉成一条可前后追踪的结构线。正向层级是：

```text
聚拢场 -> 聚拢事件 -> 聚拢单元 -> 聚拢段 -> 聚点
```

编码和结构表的分工：

| 层 | 机器要怎么理解 | 主要字段 / 编码 |
|---|---|---|
| 编码定位表 | 先在这里打点和召回；这是大库查询入口 | `variable_points.variable_type = story_progression_*` |
| 编码字典 | 看每类故事进程编码有哪些节点、覆盖多少聚点 | `variable_dictionary` 中的 `story_progression_*` |
| 容器索引 | 看某章、某场、某事件、某单元内部有哪些故事进程编码 | `container_variable_index` 中的 `story_progression_*` |
| 节点表 | 回答“这个层级在讲什么，从哪里开始，到哪里结束” | `story_progression_nodes` 的 `node_id / layer_code / raw_id / parent_node_id / start_segment_no / end_segment_no / title / summary` |
| 关系表 | 回答“谁包含谁，谁后面接谁，同一上层内部怎么往后走” | `story_progression_edges` 的 `edge_type / from_node_id / to_node_id / relation_label / atom_distance` |
| 总览表 | 回答“五层各有多少节点，覆盖到哪里” | `story_progression_layer_stats` |

故事进程轴能解决的主要问题：

1. 查一段故事在全书结构中属于哪个聚拢场、聚拢事件、聚拢单元、聚拢段。
2. 查一个聚拢场里包含哪些事件，一个事件里包含哪些单元，一个单元里包含哪些段和聚点。
3. 查同层前后：上一个/下一个聚拢场、上一个/下一个事件、同一事件内上一个/下一个单元。
4. 查故事推进：某事如何起、如何承接、哪里转折、哪里收束、后面接了什么。
5. 查长线结构：某人物、物象、空间、话语或功能点，落在哪些故事节点上，如何沿故事进程前后分布。
6. 查材料概述：快速知道某个场、事件、单元“在讲什么”，再决定是否下钻到原文。
7. 查证据边界：一个编码命中点到底只是孤立聚点，还是属于同一聚拢段、同一单元、同一事件、同一场。

基本查法：

```text
先查 variable_points 的 story_progression_* 查询编码
-> 得到聚点、聚拢段、聚拢单元、聚拢事件或聚拢场的编码命中
-> 用 variable_value_id/raw_value 对接 story_progression_nodes.node_id
-> 用 parent_child 边向上回收或向下展开
-> 用 next_global_same_layer 看全书同层后继
-> 用 next_within_parent 看同一上层内部后继
-> 需要裁判时回 start_segment_no/end_segment_no 对应原文
```

使用纪律：`story_progression_nodes.summary` 只能作结构概述和取材导航，不能替代原文；`story_progression_edges` 只能证明工程层级上的包含、前后和距离，不能自动证明文学意义。文学意义仍要交叉人物、话语、描写、时间、功能评价，再回原文裁判。

故事线时间线对齐层已经入库：`story_time_alignment_nodes / story_time_alignment_edges / story_time_alignment_layer_stats`。这不是把时间线和故事线硬合成一条线，而是在唯一中心大库里把两条线拉出来对齐。对齐方式是：故事节点范围 -> 时间锚点 / 时间区间 / 前后最近时间点 / 距离。它不做点对点硬配，允许一个聚拢场、聚拢事件或聚拢单元覆盖一个时间窗口，或夹在两个时间锚点之间。

人怎么理解：故事线回答“这段故事前后在讲什么”，时间线回答“这段故事在书内时间上靠近哪里”。对齐层回答二者之间的关系：这个故事节点是否覆盖某个时间点、是否落在某个时间区间、前后最近时间点是谁、离它们有多远。用户不需要记表名，只要说“查故事线和时间线对齐”“查这个事件在什么时间附近”“查这个时间段里有哪些故事”。

大库里怎么用：正式查询仍只进唯一中心大库。先用故事进程层定故事节点，再用故事线时间线对齐层看时间位置；如果从时间问题进入，则先定时间节点或时间区间，再反查落在其中的故事节点。

大库内部已经有三类对齐入口：`story_time_alignment_relation / story_time_alignment_span / story_time_alignment_nearest_pair`。它们分别表示：对齐关系、所在时间区间、前后最近时间锚点。查到后再接 `story_time_alignment_nodes / story_time_alignment_edges` 看具体故事节点和时间节点的关系。

基本查法：

```text
问故事在什么时间附近：故事节点 -> 对齐层 -> 时间点/时间区间 -> 原文
问某时间段有什么故事：时间点/时间区间 -> 对齐层 -> 聚拢场/事件/单元 -> 原文
问两件事时间上近不近：两个故事节点 -> 各自前后时间锚点和距离 -> 比较 -> 原文
问命运线是否同步推进：人物线索点 -> 故事进程轴排序 -> 时间对齐层校准 -> 原文
```

使用纪律：故事线时间线对齐层只能证明工程上的时间夹逼、覆盖和邻近，不自动证明真实日历时间；如果时间节点本身有复验状态，结论要继承时间节点的复验等级。它的结论说法应是“在某时间点附近/落在某时间区间/夹在两时间锚点之间”，不要直接说成“确切发生在某日”。

推进结构内部先分这些推进轴：

| 推进轴 | 主要问法 | 先用什么编码 | 怎么取证 |
|---|---|---|---|
| 时间推进 / 长线推进 | 先后、第几回到第几回、早晚、春秋、阶段变化、两点之间相隔多远、某人某事落在哪个时间段 | `time_nodes / time_spans / atom_time_projection / time_anchor_distance / time_qualities` + `chapter_no / global_atom_order` | 先查聚点的时间投影，再看前后时间锚点和夹逼区间；涉及人物、关系、话语、描写、事件时再交叉对应编码；最后回原文判断时间关系是否成立 |
| 故事线-时间线对齐 | 某个聚拢场/事件/单元处在什么时间附近；某个时间段里有哪些故事节点；故事推进和时间推进是否同步 | `story_time_alignment_nodes / story_time_alignment_edges` + 时间层 + 故事进程轴 | 先定故事节点或时间区间，再看覆盖、区间、前后锚点和距离；结论说“附近/区间/夹逼”，最后回原文 |
| 回目推进 | 某线索在哪几回出现、回目之间怎样接续 | `chapter_no / chapter_title / global_atom_order` | 按回目列点，再看每回内部的聚点、聚拢段和事件承接 |
| 事件推进 / 故事进程推进 | 一件事怎样起、怎样转、怎样收、后果是什么；几个单元如何合成事件，几个事件如何合成场 | `story_progression_nodes / story_progression_edges` + `event / event_id / scene_group_id / global_atom_order` | 先按内部层级查聚拢场、聚拢事件、聚拢单元、聚拢段、聚点，再看同层后继和父子包含；必要时交叉人物、物象、空间和功能评价 |
| 关系推进 | 两个人关系怎样变化，谁靠近谁、疏远谁、影响谁 | `person_identity / relationship_type / speech_dialogue_pair / scene_id` | 先做人名归一，再按同聚点、同场定位、话语边、关系评价逐步取证 |
| 空间推进 | 人物或物象从哪里到哪里，场景如何转移 | `space / scene_space_hint / scene_id / global_atom_order` | 先排空间点，再看人物/物象是否同场，避免只凭地名下结论 |
| 话语推进 | 某人说话前后怎样变化，两人对话线怎样发展 | `speech_* / utterance_id / old_segment_no / global_atom_order` | 先取话语点，再按说话人、受话人、原话、话语距离和上下文判断 |
| 证据推进 | 一个问题如何回到原文取证 | `variable_type / matched_text / old_segment_no / atom_code / scene_id / chapter_no` | 先由问题定编码，再交叉定位聚点；从聚点回原文窗口，摘原句和上下文，再判断是否能证明问题 |
| 功能推进 | 象征、伏笔、照应、命运提示怎样逐步显现 | `function_commentary_axis / relationship_type / emotion_symbolic_function / prophecy` | 先找功能点，再回人物、描写、事件或诗文证据，看它是不是前后相互照应 |
| 命运功能线-故事时间合流 | 命运预示、伏笔、谶语、象征、病死结果怎样沿故事和时间展开 | `fate_function_arc_points / fate_function_story_time_alignment` + 时间层 + 故事进程轴 | 先查命运功能点，再看它落在哪个聚拢事件和时间区间；它能证明“预示点/结果点在哪里”，不能自动证明应验，必须再回原文和后文事件裁判 |
| 三人个人命运-故事时间合流 | 查林黛玉、贾宝玉、薛宝钗个人命运怎样沿故事和时间流动 | `daiyu_fate_* / baoyu_fate_* / baochai_fate_*` + `story_time_alignment_nodes` | 先查个人纯命运节点，再接已合好的故事时间层，看它落在哪个聚拢事件、时间区间和前后距离；推进边只说明前后流动，不自动证明因果，最后回原文 |
| 宝黛钗三人交集点 | 查林黛玉、贾宝玉、薛宝钗三条个人命运线在哪里交汇 | `trio_fate_convergence_points / trio_fate_convergence_person_nodes / trio_fate_convergence_edges` + 三人个人命运合流层 | 先查交合点主题和强度，再看三人各自挂接的个人命运节点；同聚拢事件是基础交合，同场/同段/同聚点只代表更强交合，最后回原文裁判 |
| 宝黛钗两两交集点 | 查宝黛、宝钗、钗黛两条个人命运线在哪里交汇 | `pair_fate_convergence_points / pair_fate_convergence_person_nodes / pair_fate_convergence_edges` + 三人个人命运合流层 | 先选 pair，再查交合主题和强度；同聚拢事件是基础交合，同场/同段/同聚点只代表更强交合，最后回原文裁判 |
| 宝黛关系弧线-故事时间合流 | 宝黛关系怎样从初见、亲密、误会、诗文情意、读书分歧到婚事阴影推进 | `baodai_relationship_arc_points / baodai_story_time_alignment` + 话语层 + 时间层 + 故事进程轴 | 先查宝黛关系场，再看关系阶段、对话强度、故事事件和时间区间；阶段名是取材导航，强结论必须回宝黛原话和同场原文 |

推进结构口诀：先定一条线，再把线拆成点；先按顺序排点，再用人物、描写和原文取证裁判每个点是否成立；最后才把多个成立点合成“发展过程”。







宝黛钗个人命运专题用法：

这一组分三层读：第一层是三个人各自的个人命运故事时间合流，表为 `daiyu_fate_* / baoyu_fate_* / baochai_fate_*`；第二层是宝黛钗两两交集，表为 `pair_fate_convergence_*`，可分宝黛、宝钗、钗黛查两条个人命运线在哪里交汇；第三层是宝黛钗三人交集，表为 `trio_fate_convergence_*`，查三条线同时进入同一故事时间区域的位置。

金玉木石、婚姻压力不另开孤立入口。它已经落在宝玉的“金玉婚姻压力”、宝钗的“金玉姻缘”、黛玉的“婚事阴影”，并在两两交合和三人交合中形成主题：`金玉婚姻交合 / 木石情缘交合 / 钗黛婚事压力交合 / 金玉木石婚姻压力`。查这类题，先查个人阶段，再查两两或三人交合点，最后回原文。

例子：查“金玉和木石的婚姻压力怎样展开”，先查 `baoyu_fate_stage=金玉婚姻压力`、`baochai_fate_stage=金玉姻缘`、`daiyu_fate_stage=婚事阴影`；再查 `pair_fate_convergence_theme` 和 `trio_fate_convergence_theme` 中的婚姻压力主题；最后进入挂接节点看三个人各自在哪个聚拢事件、时间区间和原文证据中被牵入。

宝黛钗三人交集点用法：

这个层解决的不是“某一个人的命运”，而是三条个人命运线在哪里进入同一个故事时间区域。查时先用 `trio_fate_convergence_theme` 找交合主题，如情缘婚姻、诗社才情、病弱断裂、规训秩序；再用 `trio_fate_convergence_strength` 分强弱。进入某个交合点后，查 `trio_fate_convergence_person_nodes`，分别看黛玉、宝玉、宝钗各自是哪一个个人命运节点进入这里。它能快速回答“三个人的命运在哪些故事事件里交叉”，但不能自动证明三人关系因果，最后仍回原文。

三人个人命运库用法：

林黛玉、贾宝玉、薛宝钗已经形成同规格个人命运库和故事时间合流层。查个人命运时，不要重新拆故事线和时间线；先在唯一中心大库查 `daiyu_fate_stage / baoyu_fate_stage / baochai_fate_stage`，确定个人命运阶段，再查对应的 `*_fate_story_time_alignment` 看聚拢事件、时间区间、前后时间锚点和距离。需要看前后变化时查 `*_fate_progression_edges`。强结论必须回个人命运库证据点和原文。

例子：查“宝玉的命运怎样从衔玉到出家推进”，先查 `baoyu_fate_stage` 中“衔玉与命名、规训与挨打、病弱失玉、应试出家”等阶段，再按 `baoyu_fate_progression_edges` 排前后；每个节点用 `baoyu_fate_story_time_alignment` 找它所在的聚拢事件和时间区间，最后回原文裁判。

例子：查“宝钗的金玉姻缘在哪些时间故事节点出现”，先查 `baochai_fate_stage=金玉姻缘`，再查 `baochai_fate_story_time_alignment` 中对应聚拢事件、时间区间和最近时间锚点；它只能给出取材和分布，不能替代原文判断。

长线合流专用说明：命运功能线和宝黛关系弧线已经形成大库内的故事时间合流层。它们不是新的查询入口，仍只进坐标查询唯一中心大库。命运功能线用于查“预示、伏笔、谶语、象征、病死结果”这些点在故事进程和时间位置上的分布；宝黛关系弧线用于查宝黛关系场在故事进程和时间位置上的分布。两者都只能先给出取材定位和阶段导航，最后结论仍回原文裁判。

使用例：

| 问题 | 大库取法 | 注意 |
|---|---|---|
| 命运提示集中在哪些故事阶段？ | 查 `fate_function_story_time_alignment`，按聚拢事件、时间区间、功能类别汇总 | 只能说提示点分布，不能直接说已经应验 |
| 某个判词或谶语后面怎样照应？ | 先查命运功能点，再交叉后文事件和人物，按全书顺序排 | 必须回原文证明“预示”和“照应”关系 |
| 宝黛关系在哪些阶段最密集？ | 查 `baodai_story_time_alignment`，按关系阶段、故事事件、时间区间、信号强弱汇总 | 机器阶段名只作取材导航 |
| 第某个时间段里宝黛关系发生了什么？ | 先定时间区间，再反查 `baodai_story_time_alignment` 的关系场 | 回宝黛原话和同场原文裁判 |


时间长线推进专用查法：时间层已经进入坐标查询唯一中心大库。以后凡是问“什么时候、前后多久、在哪个阶段、春夏秋冬、白天夜晚、某条线怎样推进”，不要离开大库找小库，直接在唯一中心大库里查这些表和编码。

正式时间口径说明：时间查询以 `time_nodes / time_spans / atom_time_projection / time_anchor_distance / time_qualities` 为准。`season` 作为时间属性筛选，`time_block_id / time_block_label` 作为坐标辅助线索，不能替代时间主轴和距离函数。


| 时间层对象 | 大库表 / 查询编码 | 作用 |
|---|---|---|
| 主轴时间点 | `time_nodes`；编码 `time_node_projection` | 查全书内部成立的时间锚点，如某回、某事件、某阶段落在主轴哪里 |
| 时间区间 | `time_spans`；编码 `time_span_projection` | 查某聚点夹在两个时间点之间，适合模糊时间和阶段题 |
| 参照时间点 | `reference_time_points` | 放外部旁证、争议待核、工程推定参照；可以复核，不能直接替代主轴 |
| 时间属性 | `time_qualities`；可交叉 `season`，`time_block_label` 只作坐标辅助线索 | 查春秋、节令、日期、早晚昼夜等属性 |
| 聚点时间投影 | `atom_time_projection` | 每个聚点落到哪个时间点或时间区间 |
| 时间距离函数 | `time_anchor_distance`；编码 `time_prev_node / time_next_node / time_distance_relation / time_distance_label` | 查某聚点离前后时间锚点多远，是否在两点之间 |

时间推进的基本动作：

```text
自然语言问题
-> 抽出对象：人物 / 关系 / 事件 / 话语 / 描写 / 物象 / 空间
-> 抽出时间要求：先后 / 阶段 / 春秋 / 昼夜 / 两点距离 / 长线变化
-> 查唯一中心大库：对象编码 + 时间层表或时间投影编码
-> 用 global_atom_order 排序，并用 time_anchor_distance 看前后锚点
-> 回聚点原文窗口裁判
```

时间推进举例：

| 示例问题 | 编码翻译 | 查法 |
|---|---|---|
| 林黛玉全书的命运线怎样按时间推进？ | `person_identity=林黛玉` + `atom_time_projection` + `time_anchor_distance` + 必要时交叉 `event / relationship_type / speech_* / description_*` | 先取黛玉全部聚点，接时间投影，按 `global_atom_order` 和时间锚点排序；每个节点回原文裁判它是事件、对话、描写还是评价 |
| 林黛玉在春天或秋天有哪些重要出现？ | `person_identity=林黛玉` + `time_qualities.quality_type=season` 或 `variable_points.variable_type=season` | 先查季节属性，再与黛玉聚点交叉；同聚点优先，不足看同聚拢段/同场定位；回原文确认季节是否属于该场 |
| 宝玉和黛玉关系在哪些时间段发生变化？ | `person_identity=贾宝玉` + `person_identity=林黛玉` + `relationship_type / speech_dialogue_pair` + `time_spans / time_anchor_distance` | 先找两人同聚点、同场、话语边和关系评价点；再按时间主轴排序，判断每个阶段是亲近、冲突、误会、和解还是旁证 |
| 某事件离上一个明确时间点多远？ | `event` + `time_anchor_distance.prev_time_node_id / prev_atom_distance` | 先定位事件聚点，再查前后时间锚点和距离标签；只能说工程距离和文本前后距离，不能自动换算成现实天数 |
| 某段话发生在白天还是夜里？ | `speech_*` + `time_qualities.quality_type=day_part` | 先定位话语聚点，再查日内时段；若无同点日内时段，只能放宽到同聚拢段或同场，并标明放宽层级 |
| 某个聚拢事件落在哪个时间段附近？ | 先定该聚拢事件，再查故事线时间线对齐层 | 看它是否覆盖时间点、落入哪个时间区间、前后最近时间锚点是谁；回该事件起止原文裁判 |
| 某个时间区间里有哪些故事节点？ | 先定 `time_spans` 中的时间区间，再反查故事线时间线对齐层 | 列出落在该区间的聚拢场、聚拢事件、聚拢单元；按故事进程顺序排列，再回原文 |

四结构编码定位学习样例：下面的例子只说明方法，不直接产出答案。所有例子都只读坐标查询唯一中心大库，先把自然语言问题翻成查询编码，再交叉定位聚点，最后回原文裁判。

| 结构 | 示例问题 | 编码翻译 | 定位动作 | 原文裁判 |
|---|---|---|---|---|
| 人物结构 | 林黛玉和香菱有没有说过话？ | `person_identity=林黛玉` + `person_identity=香菱/英莲/秋菱` + `speech_dialogue_pair_refined` 或 `speech_dialogue_pair` | 先查人物归一，再查话语边；优先同 `utterance_id`，再看 `old_segment_no / atom_code / scene_id / event_id` | 回 `utterance_points.opening_original` 和上下文，判断是直接对话、同场说话，还是没有成立 |
| 人物结构 | 林黛玉全书和哪些人有交集？ | `person_identity=林黛玉` + 其他 `person_identity` + `scene_person_hint / relationship_type / speech_dialogue_pair` | 先取林黛玉全部聚点，再与其他人物编码求交；分同聚点、同聚拢段、同场定位、同聚拢事件、近距 | 回原文判断交集性质：同场、对话、评价、动作影响、亲疏关系，不能只凭同回下结论 |
| 人物结构 | 某人话里有没有提到某物，如黛玉说过鸡或狗吗？ | `person_identity=林黛玉` + `speech_speaker` + `speech_person_object` 或 `speech_object_term=鸡/狗/犬` | 先锁定说话人，再查话语内容物象命中；只承认同一 `utterance_id` 或同一句原话中的词 | 回本人原话 `opening_original`，确认词在她本人话里，而不是旁人说或叙述文字 |
| 描写结构 | 林黛玉全书哪些地方写过衣服？ | `person_identity=林黛玉` + `presentation_description` + `description_clothing` | 先取服饰描写聚点，再与林黛玉人物点交叉；同聚点优先，不足放宽到同 `scene_id`，并看描写对象是否指向黛玉 | 回描写原文，确认衣服描写确实落在黛玉身上，不能把同场他人衣服误算给她 |
| 描写结构 | 哪些段落写室内空间、景物或器物细节？ | `presentation_description` + `description_indoor_space / description_scenery / description_object_detail`，必要时交叉 `space / object_fullscan_*` | 先按描写子类取点，再按空间或物象编码过滤；看 `old_segment_no / atom_code / scene_id` 是否同一处 | 回原文看这段是在描写环境、景物、器物，还是只是事件经过里顺带出现的名词 |
| 描写结构 | 某人高兴或不高兴在哪里被写出来？ | `person_identity=某人` + `description_psychology_emotion`；若是本人说出来，转 `speech_*` | 先查情绪心理描写，再与人物点交叉；若命中来自话语，转入话语原话判断 | 回原文区分叙述描写、本人话语、旁人评价、同场候选 |
| 推进结构 | 某人物命运线如何从前到后变化？ | `person_identity=某人` + `chapter_no / global_atom_order` + `event / relationship_type / function_commentary_axis` | 先取人物全部聚点并排序，再按事件、关系、功能点分组；必要时按聚拢事件/聚拢场合并 | 每个节点都回原文取证，最后只把已经成立的节点连成命运推进线 |
| 推进结构 | 某关系怎样一步步变化？ | 两组 `person_identity` + `relationship_type` + `speech_dialogue_pair` + `scene_id` + `global_atom_order` | 先找两人同聚点/同场/话语边，再按全书顺序排列；看关系评价、对话、行动是否连续 | 回原文判断每个阶段是靠近、冲突、评价、影响还是只是同场 |
| 推进结构 | 某个故事节点在书内时间上靠近哪里？ | 故事进程节点 + 故事线时间线对齐层 | 先定位聚拢场/事件/单元，再查其覆盖的时间点、所在时间区间、前后最近时间锚点和距离 | 回该故事节点起止原文，并继承时间节点的证据等级；不能把附近说成确日 |
| 推进结构 | 某个时间段里发生了哪些故事？ | 时间节点或时间区间 + 故事线时间线对齐层 | 先定时间段，再反查落在其中的聚拢场、聚拢事件、聚拢单元，按故事进程排序 | 回各故事节点原文，判断它们是否真正属于这个时间段附近 |
| 推进结构 | 一首诗、判词或梦兆怎样预示后文？ | `presentation_poem_prophecy` + `poem / prophecy` + `function_commentary_axis`，再交叉后文 `event / person_identity` | 先定位诗文判词谶语聚点，再找后文可能照应点；按 `global_atom_order` 排前后 | 原文裁判必须证明“预示”和“应验/照应”的关系，不能只凭词语相似 |
| 证据结构 | 一个结论需要摘哪些原文证据？ | 先把结论拆成 `variable_type` + 对象值，如 `person_identity/object_fullscan_*/speech_*/description_*/event` | 用编码交叉找聚点，取 `old_segment_no / atom_code / scene_id / chapter_no`；同点不够再放宽 | 摘原句和上下文，标明这条证据证明什么、不能证明什么，最后决定能否入池 |
| 证据结构 | 编码命中了，但能不能算答案？ | 命中的 `variable_type / matched_text / source_table` + 坐标字段 | 回查同聚点、同聚拢段、同场定位、近距层级，记录放宽到哪一级 | 回原文窗口裁判：强证可入主证，旁证只能辅助，候选不能直接写成结论 |
| 证据结构 | 要找一句原话或一个词的出处怎么办？ | 精确词位或话语编码：`matched_text / speech_utterance / utterance_id` + `old_segment_no / atom_code` | 先定位聚点和回目，再打开原文窗口；必要时查前后聚点 | 输出回目、聚点、原句、前后文、判断理由，不只输出编码表行 |

学习样例总规则：编码负责定位，聚点负责回跳，原文负责裁判。任何题目都不能停在“查到了编码”；必须说明这个编码如何和其他编码交叉、落在哪个聚点、回到哪段原文、这段原文能够证明什么。

学习后结构转化表：读完本模板以后，进入下一步前必须在内部填写 `Codex结构转化表`。这张表不要求前台展示给用户，但必须进入 route_context，作为坐标材料门预检字段。缺此表，不进入坐标材料包。

```text
Codex结构转化表：
1. 题目中心：____
2. 选择结构：人物结构 / 描写结构 / 推进结构 / 证据结构（可多选，但必须说明主结构）
3. 查询编码：variable_type=____；对象值=____；必要定位字段=old_segment_no / atom_code / scene_id / event_id / chapter_no / global_atom_order
4. 交叉办法：编码A + 编码B；同聚点 -> 同聚拢段 -> 同场定位 -> 同聚拢事件 -> 最近距离
5. 选择理由：为什么这个问题应进这个结构，而不是其他结构
6. 下一步查证动作：先查什么，再交叉什么，最后回哪一层原文窗口裁判
```

结构转化表的作用：让机器在拿到策略令牌以后，把自然语言问题真正翻译成可执行的坐标编码。它回答三个问题：我选什么结构？我用什么查询编码？我为什么这样查？

诗文判词谶语呈现的查法：凡题目问诗、词、曲、令、联、偈、题咏、判词、梦兆、谶语、签诗、花名签、灯谜、册判、红楼梦曲，都先归到 `presentation_poem_prophecy`，再看底层是 `poem` 还是 `prophecy`，最后回原文判断它是单纯成文、命运预示，还是二者兼有。

事件叙述只细化一位即可：起事、推进、转折、结果、余波。再细就回具体 `event` 和原文，不在顶层呈现方式里继续拆。

功能评价不是顶层呈现方式。它必须附着在话语、描写、事件叙述、诗文判词谶语这些材料上使用：先确定这段是什么呈现，再看它附带什么功能判断。常见落点是关系揭示、性格揭示、象征、伏笔、照应、叙述者评价、旁人评价、命运提示、结构转场。

坐标工程结构层级锁定：

```text
聚点 -> 聚拢段 -> 聚拢单元 -> 聚拢事件 -> 聚拢场 -> 聚拢域（预留）
```

上面是从证据回跳的由小到大顺序。做故事进程、事件推进、全书结构线时，按由大到小展开：

```text
聚拢场 -> 聚拢事件 -> 聚拢单元 -> 聚拢段 -> 聚点
```

机器字段映射：

```text
聚点：old_segment_no
聚拢段：atom_code
聚拢单元：cluster_id
聚拢事件：event_id
聚拢场：scene_group_id
聚拢域：预留概念，当前未建正式字段
```

功能评价轴的建库原则：不能从聚点、聚拢段零碎直接起评。必须先从大结构定功能，再往小结构落点。顺序是：

```text
聚拢场 / 聚拢事件 / 聚拢单元
-> 先判断这个大结构为什么重要
-> 再逐层落到聚拢段 / 聚点
-> 最后用 scene_group_id / event_id / cluster_id / atom_code / old_segment_no 回原文裁判
```

层级口径锁定：功能评价轴的大结构不是 time_block。正式时间定位已经转入 `time_nodes / time_spans / atom_time_projection / time_anchor_distance / time_qualities`；`time_block_label` 只作坐标辅助线索，不作为时间推进的主查询口径。`time_block_id` 仍是聚点坐标字段之一，但不作为本轴的大义层。`scene_id` 主要用于同场定位和回跳，不等同于这里说的“聚拢场”。本轴按 `old_segment_no -> atom_code -> cluster_id -> event_id -> scene_group_id` 挂接，并预留“聚拢域”上层概念。

这样做的原因：功能评价问的是“这段为什么重要”，而重要性通常来自一个聚拢场、一组事件或一条线索，不是孤立聚点或聚拢段自己能完全决定。聚点和聚拢段只负责承接、局部证明、回原文定位。

功能评价轴固定库表：

```text
function_evaluation_unit_cards：结构功能卡。按 old_segment_no / atom_code / cluster_id / event_id / scene_group_id 挂接，聚拢域预留。
function_evaluation_cards：聚拢段附着卡。说明本段继承了哪些大结构功能。
function_evaluation_axis_points：功能命中点。查关系揭示、性格揭示、象征、伏笔线索、照应、命运预示、叙述评价等。
variable_points.variable_type='function_commentary_axis'：进入唯一中心大库的正式查询编码。
```

功能评价查询法：先查 `function_commentary_axis` 找功能类别，再回 `function_evaluation_unit_cards` 看它所在的大结构为什么成立，然后回 `function_evaluation_cards / atom_code / old_segment_no` 拿聚拢段和聚点证据。不要只凭单个聚点或聚拢段的词语直接下文学判断。

## 通用硬规则

本文件内置下列查询能力：

```text
读题 -> 判断题型 -> 拆对象 -> 定主轴 -> 人物/物象/空间归一 -> 强复合 -> 子问题队列 -> 材料升级条件 -> 回原文裁判
```

坐标工程的工作信念：坐标查询中心体系就是《红楼梦》的全书坐标总库。红楼梦里的问题，先死心塌地在这个库系里找；没有找不到的材料，只有还没想出的查询编码和交叉办法。先把问题翻成查询编码和对象值，再用两两交叉、交集、同场、最近距离定位，最后回原文裁判。

工程灵魂：用查询编码定位材料。先定编码，再找聚点；同聚点不够，再上推到聚拢段、同场定位、聚拢事件和最近距离，最后回原文裁判。

例：问“林黛玉有没有和香菱说过话”。先把问题拆成三个编码：`person_identity=林黛玉`、`person_identity=香菱/英莲/秋菱`、`speech_dialogue_pair`。先在唯一中心大库里查这三个编码有没有同一个话语点或同聚点；没有就放宽到同聚拢段、同场定位、同聚拢事件，再按 `global_atom_order` 看最近距离。查到候选后，不直接下结论，必须回 `utterance_id / old_segment_no / atom_code / scene_id / chapter_no` 对应的原话和上下文，判断是直接对话、同场说话，还是没有成立。

坐标题总分类导航：先判断题目问的是什么，也先判断它属于哪种材料呈现结构，再进入下方具体模板。不要一上来就查词，也不要先想小表；先把问题分到大类和呈现结构，再回坐标查询唯一中心大库，用大库查询编码取材。

| 大类 | 题目里常见问法 | 先入模板 | 大库查询编码（variable_type） |
|---|---|---|---|
| 查人物 | 某人在哪里、做了什么、命运如何、和谁有关 | 01 / 02 / 04 / 07 | `person_identity` |
| 查讲话 / 对话 / 话语内容 | 谁说、对谁说、有没有说过话、说了什么、怎么说、话里提到什么 | 11 | `speech_*` |
| 查物象 | 某物在哪里出现、谁接触、谁提到、有什么作用 | 03 / 04；若在话里出现转 11 | `object_fullscan_item / object_fullscan_class / object_fullscan_cooccurrence_candidate`；话里物象仍走 `speech_*` |
| 查空间场景 | 在哪里、同场有哪些人/物、某场发生什么 | 09 | `space / scene_person_hint / scene_object_hint / scene_space_hint` |
| 查时间顺序 | 第一次、先后、阶段、长线变化 | 04 | 以 `atom_time_projection / time_anchor_distance / time_nodes / time_spans` 为正式时间定位；`chapter_no / global_atom_order` 做全书顺序辅助 |
| 查事件作用 | 某事如何发生、造成什么后果、有什么影响 | 07 | `event / function / relationship_type` |
| 查诗文判词谶语 / 梦境预示 | 诗、词、曲、令、联、偈、题咏、判词、梦兆、谶语、签诗、花名签、灯谜、册判、红楼梦曲、幻境、复述边界 | 05 / 10；涉及开口转 11 | `presentation_poem_prophecy`，再交叉 `poem / prophecy / perspective` |
| 查描写 / 评价 / 情绪 | 容貌、服饰、景物、室内、室外、环境氛围、心理、动作、神情声态 | 06 / 07；若是人物说出来转 11 | `description_*`；旧线索 `emotion_symbolic_function / function / perspective` 只作辅助 |
| 复杂综合题 | 多人物、多物象、多场景、多阈值、全书长线 | 00 先拆 | 每个子问题仍回坐标查询唯一中心大库，重新选大库查询编码 |

总分类规则：人物题先找 `person_identity`，普通物象题先找 `object_fullscan_item / object_fullscan_class / object_fullscan_cooccurrence_candidate`，空间题先找 `space/scene_*`，时间题先找 `time_*`，事件题先找 `event/function`；凡是题目问到讲话、对话、话语内容、说话方式、话里提到什么，一律先进模板 11，用 `speech_*` 查询编码解决。查询编码只是入口，结论必须继续交叉 `atom_code / scene_id / global_atom_order` 并回原文。

编码轴先定，聚点直接定位：人物、空间、物象、事件等问题，第一步仍然是把题目翻成查询编码轴和对象值，例如 `person_identity=林黛玉`、`space=潇湘馆`、`object_fullscan_item=手帕`、`event=葬花`。`variable_points` 本身已经带有聚点坐标，基础定位已经完成；查询器不要另开入口，不要先找别的表。

解释映射分库归位：人物、空间、物象、事件四类解释映射分库已经归入 `00_底库登记处/26_坐标解释映射分库包/`。这些分库只在维护、复核、解释映射、将来重建对应关系时使用，按 `atom_code / scene_id / chapter_no / global_atom_order / canonical_key / matched_text` 与唯一中心大库对应。正式查询不进入这些分库；正式查询只进入坐标查询唯一中心大库，先用查询编码和聚点坐标交叉，再回原文裁判。

话语归口总则：凡是问题里出现“说、讲、问、答、回、骂、笑道、哭道、唱、命令、劝、告诉、转述、谁对谁说、说给谁听、说了什么、话里提到什么、有没有说过话、和多少人说过话、多人同场说话、在场没开口、反问、疑问、语气、物象进入话语”等意思，都先归入模板 11。模板 11 是坐标工程处理讲话、话语、对话、说话内容、说话方式的总入口。

话语查询硬判准：题目里凡是涉及讲话、对话、话语内容的咨询和查询，都在模板 11 里能够解决。先进入话语坐标，再按说话人、受话人、原话、话语边、话语距离、话语内容标注取材；不要先散入普通人物题、普通物象题或普通情绪题。

话语/对话策略提级：研究讲话本身就是《红楼梦》坐标工程的重要工程。凡是和人物开口、对话关系、话语内容、话语距离、话语句式、话语语气有关的问题，不按普通人物关系题粗查，先进入“话语/对话坐标查询策略”，再按说话人、受话人、话语边、话语邻域、逐句内容标注回原文裁判。

坐标工程八步执行法：

| 步骤 | 必做动作 | 坐标工程判准 |
|---|---|---|
| 1. 读题 | 把问题读成可查变量 | 人物、物象、空间、时间、事件、关系、诗文、说话先分清 |
| 2. 判型 | 选择 00-11 模板 | 凡是讲话、对话、话语内容、说话方式、句式语气，先入模板 11 |
| 3. 拆对象 | 复杂题拆成两两变量/对象 | 每个子问题必须能落到查询编码或聚拢段坐标 |
| 4. 定主轴 | 决定先从哪个查询编码进 | `person_identity`、`object_fullscan_*`、`space`、`event`、`speech_*` 等；时间顺序以 `chapter_no/global_atom_order` 为主 |
| 5. 归一 | 人名、物名、空间名先统一 | 别名、称谓、异写必须先归到同一编码 |
| 6. 交叉 | 用编码交集定位 | 人物编码 + 物象编码，人物编码 + 说话编码，空间编码 + 事件编码 |
| 7. 放宽 | 同聚点没有就逐级找近 | 同 utterance / old_segment_no / atom_code / scene_id / event_id，最后按 `global_atom_order` 找最近 |
| 8. 回原文 | 回原文裁判后入池 | 编码命中只是候选，必须交出原文、上下文、坐标和判断理由 |
 
复杂题必须使用模板 00 拆子问题。子问题不能只是小标题，必须变成两两可查对象，并且每个子问题重新选择本工程自己的策略。

策略卡是起手方法，不是死模板。组合、换卡、偏离都必须留理由，最后仍回原文和材料池。

## 现场修正回流与取词-策略联动硬锁

本策略层不是取词层，但它必须接收取词层的修正结果。凡用户中途围绕“子问题、拆分、取词、词网、策略、方法、模板、执行顺序、材料不够、补证、四态、放行”等焦点讨论，之后说继续、继续修正、继续完善、进入工程再执行，都必须先写 `Codex现场修正活账`，再重新选择或调整策略。

放行词如“好的 / 同意 / 继续 / 继续修正 / 继续完善 / 进入工程再执行”只表示用户允许继续，不表示可以跳过判定。触发词不是绝对词表；AI 必须根据刚才讨论的真实焦点判断本轮回到：

```text
1. 子问题策略讨论节点：问题是否要重新进入模板00，按两两交集重新设计子问题。
2. 取词-查表策略讨论节点：查询词、强复合、词角色、策略模板、查证顺序、用表收点方法是否要调整。
```

若补证门或四态门发现问题，仍必须回到上面两个根门之一。策略层必须说明：旧子问题/旧策略保留什么、替代什么、为什么换卡或组合、哪些旧材料仍可作为背景或可用证据。

最低字段：

```text

Codex回到门：____
```

| 描写段落 | `description_*` | 容貌、服饰、景物、室内、室外、环境氛围、心理、动作、器物细描、颜色感官 |
| 话语/人物对话 | `speech_*` | 谁说、对谁说、说了什么、怎么说、话里提到什么、和谁说过话、在场没开口 |

人物身份中枢新增硬规则：坐标中心大库收的是人物脚印，不是户籍册；中心正式人物查询编码已经从旧 `person` 切到 `person_identity`。全人物身份归一口径：不是只处理样例人物，而是中心 `center_person_identity_hub` 覆盖 370 人，`center_person_alias_authority` 覆盖 628 条别名/称谓/身份名，且每条别名都落到唯一 `person_code` 或带歧义规则。中心 sidecar 同时保留 `center_person_identity_hub / center_person_alias_authority / center_person_alias_conflict_rules / center_person_mention_resolution / center_person_segment_edges_enriched`；人物题可以在同一个中心库内由别名落到 `person_code`，再用 `person_identity` 回收聚点。`characters` 是户籍册，`person_segment_edges` 是脚印，`人物身份中枢库` 是认人中枢。凡是人物题、人物+物象、人物+空间、人物+话语、人物关系、人物命运题，必须先用人物身份中枢确认唯一 `person_code`；查询结果要同时保留原问称呼、原文实际称呼、归一人物、歧义状态。不得只用裸名字在 `variable_value_name` 里散搜。

描写层使用 `description_passage_cards` 和 `description_axis_points` 精查。`description_passage_cards` 是一聚拢段一条记录，挂接已有空间、人、物、功能线索；`description_axis_points` 是一条描写命中一条记录，能看触发词、描写类型、原文摘录和坐标。查询仍只读坐标查询唯一中心大库。

坐标工程基本动作：先把问题翻成查询编码和对象值，再做两两交叉；一个交集点可以回答简单题，多个交集点合成复杂题答案。若同聚点找不到，按 `utterance_id -> old_segment_no -> atom_code -> scene_id -> event_id -> global_atom_order 距离` 逐级放宽。

坐标工程专用判别法：

```text
1. 查询编码选错，后面全错；先判断题目该进 person_identity/object_fullscan/space/event/time/speech/description 哪一类。
2. 编码命中只是候选；必须继续查 atom_code、scene_id、event_id 或原文窗口。
3. 两两交叉是主法；人物 + 物象、人物 + 人物、人物 + 说话、空间 + 事件都先求交集。
4. 同聚点优先级高于同场定位，同场定位优先级高于同聚拢事件，同聚拢事件优先级高于最近距离。
5. 复杂题不是一次查完，而是几个交叉点合成一个结论。
```

坐标工程最低出材字段：

```text
variable_type / canonical_key / matched_text
chapter_no / old_segment_no / atom_code / scene_id / event_id
point_id 或 variable_id
原文摘录或原文回跳
交叉条件和放宽层级
进入坐标取材包与 04 入池凭证的记录
```

坐标工程高阶拆题法：

| 问题样式 | 先翻成什么编码 | 再做什么交叉 | 最后怎么合成 |
|---|---|---|---|
| 命运长线 | 人物编码 + 时间/事件编码 | 按 chapter_no、atom_order、scene_id 排列 | 多个坐标点合成一条人物线 |
| 两人有没有交集 | 人物A编码 + 人物B编码 | same_point/same_atom、same_scene、same_event、distance | 分成直接同聚点、同聚拢段、同场定位、同聚拢事件、未见 |
| 两人有没有说话 | 人物A编码 + 人物B编码 + `speech_dialogue_pair_refined` | `speech_dialogue_edges_refined` / `dialogue_context_edges` | 分成直接对话、对话共场、未见直接边 |
| 某人与全书谁说过话 | 人物编码 + `speech_dialogue_pair_refined` | 按 pair 另一端归并统计 | 输出对话对象表、次数、首次末次、样例原话 |
| 人物说到某物 | 人物编码 + 物词编码 + `speech_person_object` | utterance_id / atom_code 交叉 | 只承认本人原话里的词 |
| 人物怎么说 | 人物编码 + `speech_person_mood_refined` | 情态触发词 + 原话窗口 | 查哭、笑、骂、唱、问、答、吩咐、央求等 |
| 人物与物象 | 人物编码 + 物象编码 | old_segment_no / atom / scene / event / global_atom_order 交叉 | 判断同聚点、同聚拢段、同场定位、近距、背景 |
| 人物穿戴/容貌/心理/动作描写 | 人物编码 + `description_*` | old_segment_no / atom_code / scene_id 交叉，再回 `description_axis_points` | 判断某人衣服、容貌、高兴不高兴、心理动作描写 |
| 首次/顺序 | 对象编码 + 成立阈值 | 全量排序，再回原文裁判 | 第一个阈值成立坐标才是答案 |
| 多对象复杂题 | 每次只取两个编码 | 多个交集点分别裁判 | 几个小结论合成总答案 |

坐标工程对象命名规则：

```text
1. 用户原问里的名字要保留，工程内部另给 canonical_key。
2. 人物必须记录：原问称呼、原文实际称呼、归一人物、人物身份中枢 `person_code`、legacy character_code、别名依据、歧义状态。
3. 物象必须记录：原问词、object_fullscan 编码、speech_object_term 是否命中。
4. 空间必须记录：space 编码、scene_id、是否只是地点名命中。
5. 两个人的问题必须固定 person_A / person_B：A 是用户先问的人，B 是用户后问的人。
6. 凡是讲话、对话、话语内容题，先进入模板 11，再区分 A->B、B->A、同场说话、没有直接边。
7. 多人说话问题先拆 A-B、A-C、B-C，再回 scene_speech_network 或 scene_id 合成。
```

坐标工程交叉层级：

```text
第1层：查询编码命中，说明变量可能相关。
第2层：编码命中，说明对象进入候选。
第3层：两个编码同 atom_code，强相关。
第4层：两个编码同 scene_id，可判同场定位。
第5层：两个编码同 event_id，可判同一事件链。
第6层：不同聚点但 global_atom_order 接近，只能作为近距候选。
第7层：回原文摘录并写明理由，才可进入材料池。
```

坐标工程防错清单：

```text
1. 不能只看人物名命中，必须看 atom_code / scene_id。
2. 不能把同一回当作同场定位。
3. 不能把变量点相近直接当作关系成立。
4. 不能把 speech_silence 当作已经开口。
5. 不能把 speech_object_term 当作某人本人说过，必须交叉 speaker。
6. 不能只交编码，必须回 opening_original 或原文窗口。
7. 凡是问讲话、话语、对话、说话内容、说话方式，不能只按普通人物关系题处理，必须进 speech_* 查询编码。
```

## 模板 00｜坐标工程复杂题拆子问题

触发：一个问题包含多人物、多物象、多场景、多阈值、三元以上交集、全书长线。

执行：

```text
总问题 -> 拆成两两变量/对象子问题 -> 每个子问题重新选择 01-11 -> 进入坐标查询唯一中心大库 -> 输出聚点/聚拢段/变量点/距离共场 -> 04 入池凭证
```

硬规则：每个子问题必须有 Codex查询词、Codex强复合、Codex查证顺序、材料升级条件。

### 模板00的子问题策略讨论节点

模板00不是“列小标题”，而是子问题拆解策略。这里先用两两交集设计子问题；后面的 01-11 模板再用两两交集查表收点。两者是同一种思维的前后两次使用：

```text
前端拆题：总问题 -> 两两对象/变量子问题。
后端查询：子问题 -> 两个编码/变量/词位交叉成坐标点。
```

子问题拆解规则：

```text
1. 每条子问题只保留两个可交叉对象或变量：A + B。
2. A 通常是人物、主体、空间、事件、文类或时间阈值；B 通常是另一人物、物象、话语、诗词文、空间、事件、功能或原文字词。
3. 不许把“谁 + 谁 + 谁 + 什么 + 大主题”塞进同一条子问题；三元以上必须继续拆成 A-B、A-C、B-C 或 A-物、A-文类、A-场景。
4. 对象要越拆越细，直到能落到 person_identity、object_fullscan、space、event、speech_*、description_*、poem/prophecy、chapter_no/global_atom_order 或全文词位。
5. 每条子问题都必须说明交集办法：同 atom_code、同 old_segment_no、同 scene_id、同 event_id、距离最近、全文词位补漏或原文窗口裁判。
```

模板00最低产物：

```text
Codex子问题拆解策略：为什么要拆 / 为什么不拆；若拆，拆成几条。
Codex子问题：1. ____ 2. ____ 3. ____
Codex子问题两两交集表：
1. 子问题：____｜对象A：____｜对象B：____｜交集层级：atom/segment/scene/event/distance/full_text｜可落查询编码：____
2. 子问题：____｜对象A：____｜对象B：____｜交集层级：atom/segment/scene/event/distance/full_text｜可落查询编码：____
Codex子问题可查性裁判：每条子问题是否已经细到可查；未细到可查的，继续拆。
Codex子问题策略队列：每条子问题重新选择 01-11 的具体查询逻辑策略。
```

讨论口径：用户说“现在讨论子问题”时，必须显示模板00里的上述快照。若没有 `Codex子问题` 或 `Codex子问题两两交集表`，只能说“未写入/未形成”，不能现场补编成一个看似合理的旧过程。

## 模板 01｜坐标工程人物单体题

第一路径：人物身份中枢 -> 坐标查询唯一中心大库 variable_points 中的 `person_identity` 人物变量 -> 聚点/聚拢段/同场定位/聚拢事件。

升级条件：人物变量点与原文窗口能支持身份、行为、言行、命运阶段判断。

降级条件：同名误召、标题摘要、只有别名未归一。

人物 + 描写交叉法：

```text
1. 先做人名归一：如林黛玉 / 黛玉 / 林姑娘 / 颦儿。
2. 查人物查询编码：variable_type='person_identity'，得到人物所在 atom_code / scene_id。
3. 查描写查询编码：variable_type like 'description_%'，得到描写所在 atom_code / scene_id。
4. 先取同 atom_code；同聚拢段没有，再放宽到同 scene_id。
5. 回 description_axis_points 看描写类型、触发词、证据摘录。
6. 回原文窗口裁判：这个描写是否确实落在该人物身上。
```

例1：问“林黛玉在全书里穿了什么衣服，哪些地方描写她衣服？”

```text
人物编码：林黛玉。
描写查询编码：description_clothing。
第一交叉：person_identity=林黛玉 ∩ description_clothing，同 atom_code 优先。
第二交叉：若同聚拢段不够，放宽到同 scene_id，并看 description_passage_cards 的 linked_scene_persons 是否含林黛玉。
精查表：description_axis_points，取 trigger_text / evidence_excerpt。
输出：回目、atom_code、scene_id、服饰触发词、原文摘录、是否直接写黛玉衣服。
```

例2：问“林黛玉全书高兴和不高兴的地方？”

```text
人物编码：林黛玉。
描写查询编码：description_psychology_emotion。
高兴侧：trigger_text 可看 喜、欢喜、笑、含笑 等。
不高兴侧：trigger_text 可看 悲、愁、怒、怨、酸、妒、怕、慌、忧闷、烦恼、伤心、含泪 等。
第一交叉：person_identity=林黛玉 ∩ description_psychology_emotion，同 atom_code 优先。
第二交叉：同 scene_id + linked_scene_persons 含林黛玉，作为候选。
如果情绪是林黛玉自己说出来的，转入模板 11，用 speech_* 查本人原话。
输出必须分清：叙述描写中的情绪 / 本人说话中的情绪 / 同场旁人情绪候选。
```

## 模板 02｜坐标工程人物关系题

第一路径：人物 A/B 归一 -> 两组 variable_points -> same_point/same_atom/same_scene/same_event/distance。

升级条件：双方落在同一聚点、同一聚拢段、同一场景定位或同一聚拢事件，且原文窗口能证明动作、对话、观察、评价或影响。

降级条件：同回但不同场景定位、变量点相距远、只有单边命中。

关系证据包法补充（2026-06-23）：

```text
适用：两个人长期关系、关系弧线、父子/母子/祖孙/夫妻/主仆/情感/权力/规训/庇护/竞争/错位等问题。

标准路径：
1. 人物归一：先确定双方人物键，不能拆人名做字根。
2. 同段交集：从 person_segment_edges + segments 抽同段证据，得到 segment_no / chapter_no / quote。
3. 证据增强：补 evidence_edges 周边证据源，但只作周边线索，不单独作强关系结论。
4. 关系证据边：把同段证据整理成 relationship_evidence_edges，形成可投影薄边。
5. 弧线承接：读取 formal_relationship_arc，只作首版关系弧线，不替代原文裁判。
6. 原文复核：写作或强结论前，必须回 segment_no / chapter_no / quote 到原文核证。
7. 中心投影：只投影薄证据边；候选池、周边证据源、长篇判断不直接投影。

样板口径：
贾宝玉-贾政原“全量证据包”是高质量人工证据包样板；七对人物关系增强包向它看齐，但仍属于中心投影前的增强预备层。

宝黛钗例外：
贾宝玉—林黛玉、贾宝玉—薛宝钗、林黛玉—薛宝钗三条两两关系，已经退役出人物关系增强包。凡问宝黛钗三角/三边关系，先走坐标中心库 `trio_fate_convergence_*` / `pair_fate_convergence_*`，不要回到 PP-002/PP-003/PP-005 当正式入口。

硬边界：
查询、优化和取材阶段严禁改来源库。正式关系库、中心库写入必须另开写库授权门。旧本地文件不得因生成新包而自动删除。
```



## 茶酒药食方谱库查询策略

凡题目问药方、方剂、单方、单味药、丸丹、汤药、茶、酒、酒的配料、露、霜、食谱、菜谱、饮料谱、外用妆药、毒害禁忌，先归入茶酒药食层查询。正式入口仍是坐标查询唯一中心大库。要先分清：问“方名、谱、做法、配法、服法、功效”，走茶酒药食方谱；问“食物、零食、药品、茶酒实物、单味材料是什么/在哪里/谁吃用”，走茶酒药食实物。

固定查法：

1. 先查 `variable_points.variable_type in ('recipe_formula','recipe_formula_class','recipe_formula_effect')`，用 `variable_value_name` 找方谱名、类别或功效。
2. 再接 `recipe_formula_items` 看条目性质、原文摘录、确定度、边界标记；接 `recipe_formula_point_links` 取 `atom_code / scene_id / event_id / time_block_id / global_atom_order`。
3. 人物题交叉 `person_identity` 或 `speech_*`：例如“林黛玉吃过什么药”，用 `person_identity=林黛玉` + `recipe_formula_class=药方方剂` 求同聚点/同聚拢段/同场，再回原文。
4. 话语题交叉 `speech_*`：例如“谁的话里提到玫瑰露”，用 `speech_speaker` + `speech_object_term/recipe_formula`，只认同一句或同一话语点。
5. 推进题交叉时间和故事线：例如“玫瑰露案怎样发展”，用 `recipe_formula=玫瑰露` + `story_progression_* / time_* / global_atom_order` 排前后。
6. 边界：单方也算方；单味药、茶料、酒和酒的配料都算。纯杯盏盘碗不是方谱；如果器具承载明确茶酒药食名，只作“容器随饮方保留/待核”，最后必须回原文裁判。





茶酒药食全书复扫层：

- `drug_food_fullscan_material`：从第 1 回到第 120 回按 `chapters.full_text` 章节全文复扫出的茶酒药食实物，补足旧物象来源不细的问题。
- `drug_food_fullscan_class`：复扫实物类别。
- `drug_food_formula_candidate`：同一原文窗口内药、食、茶、酒连续出现，或带配制、烹调、服用、端出、赠送、功效动作时，形成方子/菜谱/饮谱候选，并回挂最近坐标聚点。

使用规则：先全书章节全文复扫找实物，再看是否形成连续组合；组合成立才升级为方谱、菜谱或饮谱。泛称命中只作候选，必须回原文裁判。

方谱卡分层：

- `recipe_formula`：方名或方谱条目，例如冷香丸、玫瑰露、茯苓霜、桂圆汤。
- `recipe_spectrum`：这个条目是否真有“谱”，以及谱文完整度。可区分“原文有谱 / 原文有用法 / 仅见名物 / 实物非方谱 / 边界待核”。
- `drug_food_fullscan_material`：全书章节全文复扫出的茶酒药食实物，例如鸡蛋、松子穰、蜜饯荔枝、人参、钩藤、茶、酒、蔷薇硝。
- `drug_food_fullscan_class`：全书复扫实物类别，例如食物/零食、药品/单味、茶酒饮品、外用妆药、毒害物。
- `drug_food_formula_candidate`：方子、菜谱、饮谱候选。

关键口径：方子要有方名和谱文；能在原文中把配制、服用、做法、功效整段搂出来的，标 `recipe_spectrum=原文有谱/原文有用法`。只有物名没有谱文的，标到 `drug_food_fullscan_material`，必要时在 `recipe_spectrum` 里标“仅见名物/实物非方谱/补标待核”。

示例：

| 问题 | 查询编码 | 交叉方法 | 回源裁判 |
|---|---|---|---|
| 黛玉全书服过哪些药？ | `person_identity=林黛玉` + `recipe_formula_class=药方方剂` | 先同聚点，再同聚拢段/同场；按 `global_atom_order` 排序 | 原文必须写黛玉服药、吐药、药饵饮食或病中用药 |
| 玫瑰露从谁到谁手里？ | `recipe_formula=玫瑰露` + `person_identity` + `event/story_progression_*` | 找所有玫瑰露聚点，按故事进程和人物交叉 | 原文确认取、给、偷、认、遮掩等动作 |
| 书中有哪些茶酒饮方？ | `recipe_formula_class=茶酒饮方` | 取 `formula_name` 后按回目/聚点排序 | 区分真饮品、酒席背景、器具承载 |
| 某段饮食为什么重要？ | `recipe_formula*` + `function_commentary_axis` | 方谱点与功能评价点交叉 | 原文判断它是病、馈赠、盗窃、关系、伏笔还是场面饮食 |

## 模板 06｜坐标工程原句/词义/语气题

升级条件：能给出回目、原句、说话人或叙述位置、前后文。

降级条件：脱离上下文解释词义。

## 模板 07｜坐标工程贡献/作用/影响题

第一路径：主体变量 -> 事件变量 -> 后果/评价变量 -> 顺序和距离。

升级条件：动作、结果、后续回响或评价形成可追踪链条。

降级条件：只同现但没有过程和后果。

## 模板 08｜坐标工程忌讳/规矩题

第一路径：规训词位 -> 人物/空间/事件变量 -> 原文窗口。

升级条件：原文说明禁止、害怕、回避、责备或后果。

降级条件：推测性性格判断、孤立情绪词。

## 模板 09｜坐标工程空间/场景题

第一路径：space 变量 -> person_identity/object_fullscan/event 变量 -> scene_id/atom_code/global_atom_order。

升级条件：地点、人物、物象、事件在同一场景或连续聚拢段中成立。

降级条件：只命中地点名，没有同场或功能证据。

## 模板 10｜坐标工程梦境/层级/醒后复述边界题

第一路径：人物变量 + 梦/醒/幻境/复述等词位 -> 原文层级窗口。

升级条件：能清楚标注梦中层级、醒后复述层级、叙述层级。

降级条件：凡是要判断“谁说、对谁说、说了什么、怎么说”，转入模板 11，再由模板 11 回原话裁判。

## 模板 11｜坐标工程话语内容与人物对话题

作用：这张表是坐标工程里专门查讲话、话语、对话和说话内容的总入口。它把《红楼梦》全书每一句可查的话，放成一个坐标点；这个点挂着人物、聚拢段、场景、受话人、原话、前后文。只要问题和讲话有关，就先在这个坐标体系里找。

总判准：题目只要问谁说、对谁说、有没有说过、和谁说过、说了什么、话里提到什么、这句话是什么句式、是什么语气、某句话附近谁在说话，都属于话语/对话坐标查询。这个模板不是补充功能，而是坐标工程处理讲话问题的主入口。

归口：谁说、对谁说、说给谁听、两个人有没有说过话、某人全书和多少人说过话、某人说话里提到什么、某人怎么说、多人同场谈话、在场没开口、传话转述，都属于本模板。

坐标工程的工作信念：坐标查询中心体系就是《红楼梦》的全书坐标总库。红楼梦里的问题，先死心塌地在这个库系里找；没有找不到的材料，只有还没想出的查询编码和交叉办法。先问：人物归一了吗？查询编码选对了吗？两个编码交叉了吗？同聚点找不到有没有放宽到同聚拢段、同场定位、同聚拢事件、最近距离？

复杂题主旋律：两两交叉。一个交集点可以回答简单题；复杂题要拆成几个两两交叉点。两个对象交叉出一个点，四个点合成一条线，几条线再合成一个大答案。

```text
人物 + 人物 -> 查两人关系/对话
人物 + 物 -> 查某人和某东西
人物 + 时间 -> 查某一阶段
人物 + 场景 -> 查同场/在场
人物 + 说话方式 -> 查怎么说
四个小结论 -> 汇成一条命运长线或关系长线
```

一眼模型：

```text
一句话 = 一个 utterance_id
一句话落在哪里 = atom_code / scene_id / chapter_no
一句话是谁说 = speaker_key / speaker_person_id / speech_speaker
一句话对谁说 = speech_addressee_refined / speech_dialogue_edges_refined

一句话怎么说 = speech_mood_refined / speech_person_mood_refined
这一条坐标点 = point_id
原话和上下文 = opening_original / context_before / context_after
```

基本动作：先把问题翻成编码，再用编码交叉定位，最后回原话。

```text
1. 人物归一：林黛玉、黛玉、林姑娘先归到同一人物；英莲、香菱、秋菱也先归一。
2. 判题型：是查谁说、两人对话、多人同场、说到某物、怎么说、在场没开口，还是传话。
3. 选编码：到 variable_points 看 variable_type。
4. 交叉定位：人物编码 + 说话类型编码 + atom_code/scene_id 做交集。
5. 逐级放宽：同一句 utterance_id -> 同聚点 old_segment_no/segment_no -> 同聚拢段 atom_code -> 同场定位 scene_id -> 同聚拢事件 event_id -> 最近距离。
6. 定位置：拿 point_id、utterance_id、old_segment_no/segment_no、atom_code、scene_id、chapter_no。
7. 回原话：到话语总库查 opening_original 和上下文。
8. 裁判：原话和上下文成立，才送材料池。
```

交叉定位的意思：

```text
查“某人说某物” = 人物编码 ∩ speech_person_object ∩ 物词编码。
查“两人说过话” = 人物A编码 ∩ 人物B编码 ∩ speech_dialogue_pair_refined。
查“某人和全书哪些人说过话” = 人物编码 ∩ speech_dialogue_pair_refined，按 pair 归并。
查“某人全书和多少人说过话” = 人物编码 ∩ speech_dialogue_pair_refined，统计不同对话对象。
查“多人同场谈话” = 人物组编码 ∩ scene_id ∩ speech_dialogue_pair。
查“某人怎么说” = 人物编码 ∩ speech_person_mood_refined。
查“最近/靠近/同场” = 先同 utterance/old_segment_no/atom_code/scene_id，后按 global_atom_order 距离找最近。
```

能照着做的例子：

```text
例1：问“林黛玉有没有和香菱说过话？”
- 人物编码：林黛玉；香菱/英莲。
- 先查 variable_type=speech_dialogue_pair_refined。
- 再回 speech_dialogue_edges_refined 看有没有严格 A -> B 或 B -> A。
- 如果没有，查 dialogue_context_edges，看是否同聚拢段或同场对话。
- 输出必须分三类：直接对话 / 对话共场 / 未见直接边。

例2：问“林黛玉说话里有没有提到鸡或狗？”
- 人物编码：林黛玉。
- 词编码：鸡、狗、犬。
- 查 variable_type=speech_person_object 或 speech_object_term。
- 回 utterance_object_intersections 和 utterance_points。
- 只算 opening_original 里她本人说出的词；旁人说、叙述背景、标题字不算。

例3：问“香菱/英莲从第1回到第120回和谁说话，说了什么？”
- 人物编码：英莲/香菱/秋菱归一。
- 查 variable_type=speech_speaker。
- 按 chapter_no + global_atom_order 排。
- 每条交出：回目、atom_code、scene_id、对谁说、原话、前后文。
- 这就是人物说话长线，后面才能谈命运流转。

例4：问“某人全书和多少人说过话，分别是谁？”
- 人物编码：先归一，如林黛玉/黛玉/林姑娘。
- 先查 variable_type=speech_dialogue_pair_refined。
- 用 pair 里另一端作为“对话对象”，按人物去重统计。
- 再按 chapter_no + global_atom_order 排出每个对象的首次、末次和样例原话。
- 输出必须分清：直接话语边、同场对话线索、在场未开口候选。

例5：问“三个人同场谈话，谁说、谁听、谁在场？”
- 人物组先归一。
- 查 scene_id，再查 scene_speech_network 和 dialogue_context_edges。
- 同一 scene 内列 speaker、addressee、sample_openings。
- silent_presence_points 只能说明在场没开口候选，不能直接当结论。

例6：问“王熙凤什么时候骂、命令、笑着说？”
- 人物编码：王熙凤。
- 查 variable_type=speech_person_mood_refined 或 speech_mood_refined。
- 回 speech_utterance_mood_refined，看 trigger_text、source_context、match_basis。
- 最后必须回 opening_original，确认这个说话方式确实挂在王熙凤这句话上。
```

先走唯一大库，再用查询编码和同库精查表：

话语距离/邻域固定层：

```text

speech_pair_scene_density：查两个人全书哪些场景对谈最密。

speech_neighborhood_points：查某一句话附近每一句具体是谁说、距离中心多远。
```

话语距离层只负责找场面和候选，不直接写文学判断；找到后仍回原话和上下文裁判。

话语内容逐句达标层：

```text
speech_content_annotation_cards：一话一条记录，11559 句全部 checked；查每句话是否含物象、情绪、颜色/感官、动作、评价、句式、语气、关系功能、确定性。
speech_content_axis_points：一命中一条记录；查具体轴类、词、位置、置信度、来源方法。
speech_content_axis_lexicon：本次逐句达标使用的词表。
```

话语内容层解决的是“话里有什么”：某人说话里是否提到扇子、手帕、鸡、狗、花；是否出现悲伤、责备、请求、命令；一句话是疑问、反问、陈述、祈使还是感叹。命中后仍回 opening_original 和上下文裁判。

话语题先看 `variable_points.variable_type`：

```text
speech_speaker        谁开口
speech_addressee      对谁说
speech_addressee_refined  精修后的对谁说
speech_dialogue_pair  两人对话共场
speech_dialogue_pair_refined  精修后的两人话语边
speech_object_term    说话里提到的词
speech_person_object  某人说话里提到某词
speech_mood           怎么说
speech_mood_refined   精修后的怎么说
speech_person_mood    某人怎么说
speech_person_mood_refined  精修后的某人怎么说
speech_silence        在场但没开口
speech_utterance      真实开口点
speech_content_annotation_cards  逐句内容达标卡
speech_content_axis_points  逐句内容轴命中
```

三个最常用的距离查询：

```text
1. 查某人哪里说话最密：
   speech_person_scene_density
   where speaker_key=人物名
   order by utterance_count desc

2. 查两个人哪里对谈最密：
   speech_pair_scene_density
   where person_a_name/person_b_name 命中两个人
   order by direct_edge_count desc, total_signal_count desc

3. 查某句话附近几个人说话：
   speech_neighborhood_summary
   where center_utterance_id=目标话语ID
   看 same_atom、same_scene、atom_radius_1 三层。
```

三个最常用的内容查询：

```text
1. 查某人话里提到什么物象：
   speech_content_axis_points
   where speaker_key=人物名 and axis_type in ('object','object_candidate')
   object_candidate 只作补漏候选，结论必须回原话。

2. 查某人哪些话是疑问、反问、命令、请求：
   speech_content_annotation_cards
   where speaker_key=人物名
   再看 sentence_type_primary 和 tone_type_primary。

3. 查某一句话的内容结构：
   speech_content_annotation_cards + speech_content_axis_points
   where utterance_id=目标话语ID
   输出物象、情绪、动作、评价、句式、语气和原话。
```

人物话语关系网查询法：

```text
1. 先进入人物身份中枢库，定唯一 person_code；所有别名、称谓、身份名、原文实际称呼都归到这个身份。
2. 查 variable_points 中的 speech_dialogue_pair_refined。
3. 取 pair 的另一端，得到“这个人和谁说过话”。
4. 按另一端人物去重，得到“和多少人说过话”。
5. 按 chapter_no/global_atom_order 排序，得到从前到后的关系发展。
6. 每一条边回 speech_dialogue_edges_refined，看是谁说、对谁说、哪一句。
7. 需要原话时回 utterance_id / atom_code / opening_original。
8. 需要命运长线时，把每个对象的首次、末次、高频场景、关键原话合成一条线。
```

## 全书名物序列与流转库查询策略

凡题目问全书有多少名物、某物第几次出现、全书第几个物是什么、某个具体东西如何流转，进入坐标查询唯一中心大库的 `nominal_*` 查询编码：

- `nominal_object_item`：每一次名物出现，带全书名物序号和聚点坐标。
- `nominal_object_category`：名物大类，如器物、服饰、茶酒药食、动植物、文书信物、光影声音气味。
- `nominal_object_relation`：原文词和主物象的关系，如本体、部位、影、光、声音、气味、方谱。
- `nominal_single_item_flow`：通灵宝玉、风月宝鉴、金锁、金麒麟、旧帕、诗稿、荐信等单件名物的流转候选。
- `nominal_single_item_formal_flow`：已经人机确认并回原文复核的正式单件流转点。当前林如海荐信已完成“两点一回响”；通灵宝玉已完成“顽石起点-石化-衔玉-观玉-救治-借看-怒摔-失玉-假玉排除-送回-欲还-归本处”正式样板；风月宝鉴已完成“授镜-规则-反面骷髅-正面幻象-铁锁身亡-道人收回”正式样板；金麒麟、金锁、宝黛旧帕、黛玉诗稿已完成正式样板。其他单件物仍须逐个确认后才能进入此表。
- `nominal_occurrence_distance`：每个名物点的前后名物、同一名物前后点、相隔聚点数，以及是否同聚拢段、同聚拢单元、同聚拢事件、同场、同时间块。

写文章时可直接调用已成形的正式流转线：林如海荐信、通灵宝玉、风月宝鉴、金麒麟、金锁、宝黛旧帕、黛玉诗稿；查询时统一查 `nominal_single_item_formal_flow`，再按聚点回原文取证。

使用方法：查数量先看 `nominal_objects / nominal_occurrences`；查次序按 `nominal_sequence_no`；查某物第几次出现按 `canonical_name + object_occurrence_no`；查单件流转先用 `nominal_single_item_flow` 找候选，再看 `nominal_single_item_formal_flow` 是否已有正式打标；没有正式打标时，必须按作品/问答/原文复核流程逐个裁判，不得把候选点当最终结论。查距离和所在层级按 `nominal_occurrence_distance`，再回原文裁判持有人、状态和动作。

例：查“通灵宝玉怎么流转”，先在 `nominal_single_item_flow` 看明文候选，再查 `nominal_single_item_formal_flow where item_name='通灵宝玉'`。正式表会把第95-96回假玉标成“误认排除点”，所以不会把假玉误当真玉。

例：查“风月宝鉴怎么流转”，查 `nominal_single_item_formal_flow where item_name='风月宝鉴'`。正式表会把“只照背面”标成规则点，把“照正面见凤姐”标成误用显像点，把“谁叫你们瞧正面了”标成误用裁判和收回闭环。



例：查“金麒麟怎么形成伏线”，查 `nominal_single_item_formal_flow where item_name='金麒麟'`。正式表会把黛玉冷讽、湘云佩带、拾得大金麒麟、第49回并称、第83回周瑞家的回补分成不同性质的流转/回响点。

例：查“金锁为什么压向金玉良缘”，查 `nominal_single_item_formal_flow where item_name='金锁'`。正式表会保留第8回金锁与通灵宝玉互看、第28回有玉方婚姻、第84回天配话、第96回冲喜调用，并排除第18回匾联误命中。

例：查“宝黛旧帕怎么走”，查 `nominal_single_item_formal_flow where item_name='宝黛旧帕'`。正式表会只保留宝玉给、晴雯送、黛玉题诗、旧帕重现、临终焚毁、紫鹃转述，不把小红手帕或普通手帕混进来。

例：查“黛玉诗稿怎么成为终局文字物”，查 `nominal_single_item_formal_flow where item_name='黛玉诗稿'`。正式表会保留索诗本、焚稿、紫鹃转述和宝玉后文追问。

## 全书人物关系网中心表查询策略

凡题目问“某人的父母是谁、丫鬟是谁、夫妻/姻亲是谁、师生是谁、医患是谁、官媒是谁、某个人周边关系网是什么”，进入坐标查询唯一中心大库的人物关系网表。人物关系网的 `relation_type` 只能写真实关系名，不能写话语、同场、命运线、证据包、直线接头这类工程词。若要把人物小圈接入全书大网，必须把小圈里的每一个人轮番与大网人物兑接，最后落成具体关系名。

- `person_relation_network_edges`：人物与人物之间的真实关系边，含父子、父女、母子、母女、祖孙、夫妻、恋爱/情感相属、婚姻/议亲、丫鬟与主人、小厮与主人、奶娘乳母、师生、医患、官媒、宗族、世交书信、案件证人等。
- `person_relation_network_person_summary`：每个人一行的周边关系摘要，370 人都可查。
- `person_relation_network_relation_types`：关系类型说明。
- `v_person_relation_ego_network`：按一个人物展开他的全部周边关系。
- `v_person_relation_strong_answer_edges`：只看父母、丫鬟、夫妻、师生、医患、官媒、宗族、恋爱、婚姻等可直接回答问题的强关系。

当前已建成：`370` 个人物节点，`664` 条真实关系边。查“林黛玉的父亲/母亲/丫鬟/宝黛关系”，先查 `v_person_relation_ego_network where center_person_name='林黛玉'`，需要可直接回答的强答案时查 `v_person_relation_strong_answer_edges`。

证据口径必须分清：
- `single_fact`：一句话或一个事实可证，如父亲、母亲、丫鬟、塾师；必须有 `atom_code` 和原文摘录。
- `full_arc_package`：全文关系包，如宝黛关系、宝钗黛玉关系、袭人宝玉关系；但写入表时 `relation_type` 仍必须落成恋爱、婚姻/议亲、闺阁同伴、丫鬟与主人等真实关系名。
- `person_relation_direct`：断圈补链的直接人物关系边。做法是逐人兑大网，但写入表时必须落成具体关系名，如勋贵旧交、寺庙住持、医患、管事与主家、官媒、案件证人、世交书信、宗族；不收抽象交往。
- `candidate_only`：身份圈或机器线索，没有逐条回原文，只能作为候选，不能直接写成结论。

注意：`话语对话`、`同场共现` 只作为查找线索留在各自专项库，不作为人物关系网的最终 `relation_type`。
