## 二、策略门思考说明表

### 1. 策略门的作用

策略门不是答案生成器。它的作用是让机器在进大库前，先把题目想清楚：

```text
题目问什么
-> 属于哪种查询结构
-> 应该用哪些查询编码
-> 编码之间怎样交叉
-> 交叉后落到哪个聚点、聚拢段、现场
-> 最后回哪段原文裁判
```

一句话：

```text
策略门负责把自然语言问题翻译成红楼梦人工智能咨询工程能执行的查询路线。
```

### 2. 读完策略门后的必填动作

读完策略门后，必须内部填写 `Codex结构转化表`。这张表不一定展示给用户，但必须进入工程上下文，供材料门预检。

| 字段 | 要回答的问题 | 示例 |
|---|---|---|
| 题目中心 | 这个问题真正要查什么 | 查林黛玉是否和香菱说过话 |
| 选择结构 | 先进哪条结构 | 人物结构，交叉话语对话 |
| 查询编码 | 用哪些大库编码 | `person=林黛玉` + `person=香菱/英莲/秋菱` + `speech_dialogue_pair` |
| 交叉办法 | 编码怎样碰到一起 | 同一句、同聚点、同场定位、最近距离 |
| 选择理由 | 为什么这样查 | 题目核心是两个人是否发生话语交集 |
| 下一步查证动作 | 接下来去哪里取证 | 进唯一中心大库收点，再回原文窗口裁判 |

### 3. 策略门工作原理表

| 步骤 | 工作动作 | 机器要形成的判断 | 不能停在什么地方 |
|---|---|---|---|
| 1 | 读题中心 | 判断题目是人物、描写、推进、证据，还是复合题 | 不能只抓关键词 |
| 2 | 选主结构 | 在人物结构、描写结构、推进结构、证据结构中选主入口 | 不能四处散查 |
| 3 | 拆子问题 | 复杂题拆成多个可验证小题 | 不能一次性写结论 |
| 4 | 选查询编码 | 把题目词变成 `variable_type / matched_text / canonical_key` | 不能只靠自然语言想象 |
| 5 | 做对象归一 | 人名、物名、空间名、话语对象先归一 | 不能漏别名 |
| 6 | 交叉定位 | 用多个编码求交集，按近远层级找点 | 不能只看同回 |
| 7 | 定位坐标 | 锁 `old_segment_no / atom_code / scene_id / event_id / chapter_no / global_atom_order` | 不能只有候选词 |
| 8 | 回原文裁判 | 打开聚点和上下文，判断证据是否真能证明问题 | 不能把编码命中直接当答案 |
| 9 | 入材料池 | 只把经过原文裁判的材料入池 | 不能把待复核候选混成主证 |

### 4. 四种结构的起手判断

| 结构 | 适合什么题 | 常用编码 | 核心方法 |
|---|---|---|---|
| 人物结构 | 查某人、谁和谁、人物关系、人物与物/话/空间/事件 | `person / relationship_type / speech_* / scene_person_hint` | 人物归一后，用人物编码和对象编码交叉 |
| 描写结构 | 查衣服、容貌、景物、空间环境、心理、动作、神情声态 | `presentation_description / description_* / object / space / person` | 先定描写类型，再交叉描写对象 |
| 推进结构 | 查时间推进、关系变化、事件发展、命运长线 | `chapter_no / global_atom_order / time_axis / event / relationship_type` | 把线拆成点，按顺序排，再逐点回原文 |
| 证据结构 | 查原文证据在哪里、哪些材料能证明结论 | `variable_type / matched_text / old_segment_no / atom_code / scene_id` | 由问题反推编码，再回原文摘证 |

### 5. 编码定位的主口诀

```text
编码负责定位。
聚点负责回跳。
交叉负责筛选。
原文负责裁判。
```

常用交叉顺序：

```text
同一句 / 同一话语
-> 同聚点 old_segment_no
-> 同聚拢段 atom_code
-> 同场 scene_id
-> 同聚拢事件 event_id
-> 同时间块 time_block_id
-> 最近距离 global_atom_order
```

## 三、120回时间轴设计

### 0. 外部年谱参照层

网络上已有《红楼梦年谱》类资料，可作为时间轴设计的参照底座，但不能直接当成工程事实。

本轮已下载公开网页原件并放入：

`outputs/红楼梦人工智能咨询工程策略门与时间轴设计_20260623/外部公开资料/`

本轮已进一步把该资料转成两个工作表：

| 工作表 | 当前数量 | 用途 |
|---|---:|---|
| `红楼梦年谱_wikipedia_外部年份模型.csv` | 201 条正文外部节点 | 作为外部年份模型和复核队列 |
| `红楼梦年谱_wikipedia_与现有time_axis候选对齐.csv` | 281 条候选对齐 | 试探外部节点与现有时间轴的可能关系，只作召回 |

外部年谱的特点：

| 特点 | 对本工程的启发 |
|---|---|
| 按“元年、三年、四年……十九年”组织全书大事 | 可以提供一个外部相对年序模型 |
| 把关键事件挂到月份、日期、节令 | 可以帮助发现原文时间锚点和待核节点 |
| 同一事件常有不同年龄、不同年份说法 | 说明外部年谱必须进入争议/校准层，不能压倒原文 |
| 前八十回和后四十回分开处理 | 本工程也应保留版本段落和叙事层差异 |

因此，本工程新增一个外部旁路参照口径：

```text
主轴 = 原文时间层 + 工程推论层
旁路 = 外部公开资料复核队列
```

外部年谱只做三件事：

1. 给全书相对年份提供候选模型。
2. 帮助检查本工程漏掉的硬时间锚。
3. 把争议年份、争议年龄、争议日期写入复核表。

本轮矩阵法结果：

```text
外部正文节点：201
第一轮可进入复验队列：39
第二轮新增：7
第三轮夹逼新增：3
三轮合计进入复验队列：49
旁路保留/暂不用：152
```

硬边界：

```text
外部时间节点不能进入主轴。
外部年份不能写成主轴年份。
外部资料只能反向提示：哪里需要回原文复核。
```

它不能做三件事：

1. 不能直接替代原文。
2. 不能把外部推断改写成“书中明说”。
3. 不能强迫全书服从某一家年谱。

### 1. 时间轴工程的目标

目标不是强行说全书一定经过三年或四年，而是建立一条可查询、可回源、可分层的时间轴：

```text
每个聚点
-> 属于哪个回内时间块
-> 有没有原文明示时间
-> 如果没有，工程怎样推定前后
-> 推定依据是什么
-> 置信度是多少
-> 最后能回到哪段原文
```

也就是说，以后任何人物、事件、话语、描写、物象，都能回到时间轴上。

### 2. 时间不能混成一层

全书时间必须分两大层：

| 层 | 含义 | 可不可以当真实时间 |
|---|---|---|
| 真实时间层 | 原文明示或强上下文直接支持的时间 | 可以，但要带原文证据 |
| 推论时间层 | 工程根据前后顺序、场面、事件承接推出来的时间 | 可以用于排序，不能冒充原文明示 |

同时保留两类特殊层：

| 特殊层 | 含义 | 用法 |
|---|---|---|
| 神话梦幻预示时间 | 太虚、梦境、判词、谶语、异兆、应验 | 不和现实年月硬并入同一层 |
| 机制时间 | 起点、加压、爆发、收束、应验等过程阶段 | 用来解释命运线、败落线、关系线 |

### 3. 证据等级

建议沿用已有 `time_axis.evidence_level`，并把口径写死：

| 等级 | 名称 | 判定口径 |
|---|---|---|
| A | 原文明示硬锚点 | 原文直接给出日期、节令、时辰、早晚、次日等 |
| B | 原文上下文强推 | 同一连续现场内，由“次日、过了几日、不多时”等可直接推出 |
| C | 工程顺序推定 | 根据回目、场面、事件承接、聚点顺序推断 |
| D | 外部旁证 | 来自外部年表或研究模型，只作旁路参考 |
| E | 待复核/冲突 | 时间表述互相打架或证据不足 |

关键规则：

```text
A/B 可以进入真实时间层。
C/D/E 必须留在推论层或复核层。
```

### 4. 时间轴主表建议字段

建议新建“全书时间轴节点表”，不是替代现有表，而是把已有时间底座、时间块、聚点坐标汇合起来。

| 字段 | 说明 |
|---|---|
| `timeline_node_id` | 全书时间轴节点编号 |
| `chapter_no` | 第几回 |
| `time_block_id` | 回内时间块编号 |
| `scene_group_id` | 场组编号 |
| `scene_id` | 场点/现场编号 |
| `event_id` | 聚拢事件编号 |
| `atom_code_start` / `atom_code_end` | 覆盖的聚拢段范围 |
| `old_segment_no_start` / `old_segment_no_end` | 覆盖的聚点范围 |
| `global_order_start` / `global_order_end` | 全书顺序范围 |
| `real_time_label` | 原文明示时间，如“八月十五”“五更”“次日” |
| `real_time_evidence_level` | A/B/C/D/E |
| `real_time_quote` | 原文证据 |
| `inferred_year_index` | 工程推定第几年 |
| `inferred_season` | 工程推定季节 |
| `inferred_stage` | 工程推定阶段 |
| `time_layer` | 真实时间 / 推论时间 / 梦幻预示时间 / 机制时间 |
| `inference_basis` | 推论依据 |
| `confidence` | 置信度 |
| `conflict_note` | 矛盾或疑点 |
| `review_status` | 已核 / 待核 / 争议 |

### 5. 聚点时间投影表

为了让每一件事都回到时间轴，必须有一张边表：

| 字段 | 说明 |
|---|---|
| `old_segment_no` | 聚点 |
| `atom_code` | 聚拢段 |
| `timeline_node_id` | 所属时间轴节点 |
| `time_block_id` | 原有回内时间块 |
| `time_role` | 锚点 / 覆盖 / 推论补点 / 争议点 |
| `distance_to_anchor` | 距离最近时间硬锚点多远 |
| `source_rule` | 原文明示 / 上下文顺承 / 回内排序 / 人工复核 |

这张表的作用很大：

```text
查人物，可以知道他的出场时间线。
查对话，可以知道这句话发生在时间轴哪一段。
查物象，可以知道物象在哪些时间段显影。
查事件，可以知道事件前后顺序和时间证据强弱。
```

### 6. 建设流程

| 步骤 | 动作 | 产出 |
|---|---|---|
| 1 | 以 `atom_coordinates` 全量聚点为底 | 得到 2706 个聚点的现有坐标和 `time_block_id` |
| 2 | 接入现有 `time_blocks` | 得到每回内部时间块、起止段号、证据、置信度 |
| 3 | 接入现有 `time_axis` | 得到 282 个已有时间锚和时间窗口 |
| 4 | 回原文扫描时间词 | 抽取年、月、日、时辰、节令、春夏秋冬、早晚、次日、过几日 |
| 5 | 建真实时间层 | 只收 A/B 级时间，保留原文 quote |
| 6 | 建推论时间层 | 根据时间块顺序、场面切断、事件承接推定前后 |
| 7 | 标冲突和疑点 | 不强行抹平矛盾，写入 E 级复核 |
| 8 | 给每个聚点投时间节点 | 形成 `old_segment_no -> timeline_node_id` |
| 9 | 写入坐标唯一中心大库投影 | 形成可查编码，如 `time_axis / time_point / time_block / timeline_node` |
| 10 | 抽样复验 | 查人物线、事件线、对话线能否按时间稳定排序 |

### 7. 关于“三年还是四年”

这个问题不能在开工前硬定。正确做法是：

```text
先设相对原点 T0。
再让每个明确时间锚和推论时间块落到 T0 之后的相对位置。
最后由全书节点自然推导出主线经过几年。
```

建议字段：

| 字段 | 用法 |
|---|---|
| `relative_time_order` | 全书相对顺序 |
| `relative_year_index` | 工程推定第几年 |
| `year_model` | 使用哪个年表模型 |
| `year_model_status` | 原文明示 / 工程推定 / 外部旁证 / 争议 |

这样可以同时容纳：

```text
原文明示时间
工程推论时间
外部年表模型
争议时间
```

不会把推论时间误写成原文事实。

## 四、时间轴查询能解决什么

| 问题 | 时间轴建成后的查法 |
|---|---|
| 林黛玉一生重要节点怎样推进 | `person=林黛玉` 交叉 `timeline_node`，按时间节点排序 |
| 某件物象什么时候出现、什么时候消失 | `object=某物` 交叉 `timeline_node` |
| 某两个人关系在哪些时间段发生变化 | 两个 `person` 交叉 `relationship_type / speech_* / timeline_node` |
| 某场对话发生在同日、次日还是若干日后 | `speech_*` 回到聚点，再查 `timeline_node` 和最近硬锚点 |
| 某个事件是原文明确时间还是工程推定时间 | 查 `real_time_evidence_level` 和 `inference_basis` |
| 全书哪些地方时间有矛盾 | 查 `conflict_note` 和 `review_status=争议/待核` |

## 六、季节昼夜等时间属性的归位

已核到现有时间底座：

| 位置 | 已有内容 | 说明 |
|---|---|---|
| 语义聚拢中心库 `time_axis` | `season / lunar_term / date_label / day_part` | 已经把季节、节令、日期、日内时段放在时间轴表里 |
| 坐标查询唯一中心大库 `variable_points` | `season / time_axis / time_block_label` | 已经把季节和时间轴投成可查编码 |
| 场面时间库 `time_blocks` | `time_label / time_evidence / start_segment_no / end_segment_no` | 已经有回内时间段和原文段范围 |

结论：不要另起一个独立“季节早晚库”。时间只建一个库，但库内分两种东西：

```text
主轴时间 = 什么时候发生，负责排序和定位。
时间属性 = 这个时间点/时间段带有什么季节、节令、早晚、昼夜、天气氛围。
```

推荐增加一张属性表，而不是另起一库：

```text
time_qualities
```

它挂在 `time_nodes / time_spans / atom_time_projection` 后面。

建议字段：

| 字段 | 说明 |
|---|---|
| `quality_id` | 时间属性编号 |
| `target_type` | node / span / atom |
| `target_id` | 对应的时间点、时间段或聚点 |
| `quality_type` | season / lunar_term / date_label / day_part / weather / atmosphere |
| `quality_value` | 春、暮春、秋夜、五更、月夜、炎夏永昼等 |
| `evidence_level` | 原文明示 / 上下文推定 / 工程推定 / 外部旁证 / 待判 |
| `source_quote` | 原文证据 |
| `confidence` | high / medium / low |
| `usage_rule` | 可作定位 / 可作筛选 / 只作氛围 / 待复核 |

这样查询时不会混乱：

```text
查“第几年前后” -> 查 time_nodes / time_spans。
查“春天发生了什么” -> 查 time_qualities.quality_type=season。
查“夜里发生了什么” -> 查 time_qualities.quality_type=day_part。
查“秋夜黛玉相关内容” -> season/day_part 与 person 交叉，再回聚点。
```

最终结构应为一个时间库五张表：

| 表 | 作用 |
|---|---|
| `time_nodes` | 主轴确定时间点 |
| `time_spans` | 两个确定点之间的夹逼时间段 |
| `atom_time_projection` | 聚点落到时间点/时间段 |
| `reference_time_points` | 参照时间点，放外部旁证、争议、待核 |
| `time_qualities` | 季节、节令、日内时段、天气氛围等时间属性 |

一句话：

```text
不要分两个库；做一个时间库。
春秋早晚不是另一个时间轴，而是挂在时间点、时间段和聚点上的时间属性。
```
