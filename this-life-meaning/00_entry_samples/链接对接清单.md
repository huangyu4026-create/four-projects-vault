## 三、召回闭环现状

现在已经接上的链路：

```text
页面提交问题
  -> 本地服务生成 request_id
  -> 写入 课堂收件箱
  -> 写入 召回队列
  -> 页面显示“待 Codex 处理”
  -> Codex 处理队列并写入轻量回显 JSON
  -> 页面按 request_id 读取结果并显示
```

需要注意：

```text
页面不能直接调用当前 Codex 聊天窗口。
它只能把问题写入本地队列。
Codex 必须读取队列并写入结果文件，页面才能显示正式答案。
```

因此，真正的触发词是：

```text
处理此生何为课堂队列
```

## 四、小方格用途

主讲区标题下面的小方格已经改为课程进度方格，不再是装饰。

| 方格 | 含义 |
|---|---|
| 已完成 | 显示已经完成并归档的课程 |
| 当前打开 | 显示主讲区正在看的课程 |
| 下一步 | 显示下一讲或收束动作 |
| 第一章进度 | 显示当前打开到第一章十讲中的哪一步 |

点击每个进度方格，右侧“轻量召回”区域都会显示文字说明。

## 五、回显规则

页面不是 AI 大脑，只负责课堂现场与资料回显。

每个入口至少有两层：

1. 页面内回显：在右侧轻量召回区说明这个入口是什么、接到哪里、何时调用。
2. 打开原文件：直接进入对应 Markdown、Python 服务或启动文件。

正式课堂问答、追问、总结、下课整理仍由 Codex 根据工程底座完成。
