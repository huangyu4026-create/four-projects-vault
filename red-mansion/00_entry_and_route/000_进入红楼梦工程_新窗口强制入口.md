## 一句话定案

任何新窗口、任何新线程、任何新对话，只要用户说：

```text
进入红楼梦工程
```

或等价表达：

```text
请进入红楼梦工程
用红楼梦工程查
按聚龙法查
正式查询
深度研究
```

第一步必须先走本文件规定的入口。

不允许先自由搜索 SQLite。
不允许先自由搜索全文。
不允许先凭印象回答。
不允许先走旧八步、旧九步、旧搜索词网络。
不允许把“可直接命中”当作“可直接写答案”。

新增总铁律：

```text
命中只说明材料找到了。
命中不说明材料已经裁判。
没有聚拢裁判，就没有最终答案。
```

新增机器入口铁律：

```text
先建本题闭环包，再查底层检索库。
先有聚拢库取材单，再把命中材料送入材料池。
直接查 formal_honglou_search.sqlite / segments / search_documents，不算进入聚拢库。
```

内容查询题默认先执行：

```text
python3 work/formal_honglou_closed_loop.py --question "用户原问题"
```

只有本题输出包里已经生成并读取以下文件，才算真正进入聚拢库/聚拢场：

```text
00AC_聚拢库总入口流程锁.md
00AG_聚拢库取材单.md
00AI_聚拢入材料池清单.csv
00AM_聚拢裁判首读材料池.md
```

这四件套只证明“已进入聚拢库 / 聚拢场”，不证明可以写最终答案。

最终答案另有硬验收，必须同时具备：

```text
00I_Codex材料池判定_<request_id>.md/json
00L_Codex精读材料词_<request_id>.md/json
00M_Codex写作前原文通读摘抄_<request_id>.md/json
```

`00ZI_Codex精读材料词_待生成.md` 和 `00ZP_Codex写作前原文通读摘抄_待生成.md` 只是人读同步稿位；它们可以显示内容，但不能单独替代正式 `00L/00M` 的 md+json 成品。

缺任一项，状态只能写：

```text
最终答案硬阻断：召回已命中，待 00I/00L/00M 闭环。
```

如果聚拢四件套缺任何一个，状态只能写：

```text
阻断：未进入聚拢库/材料池。
```


## 零点五、机器自检与状态词

新窗口不准用一句“已进入红楼梦工程”掩盖没有进聚拢库的问题。

状态词只能这样用：

```text
已接入红楼梦工程入口 = 已识别工程根目录，并已读 000。
已建本题闭环包 = 已运行或确认 formal_honglou_closed_loop.py 的本题 package_dir。
已进入聚拢库 / 聚拢场 = 本题 package_dir 内 00AC/00AG/00AI/00AM 已生成并已读。
召回已命中 = 只说明材料被找到，不说明可写答案。
可写最终答案 = 聚拢裁判、材料池四态、写作前原文追证摘抄均完成。
```

内容回答前必须先自检：

```text
package_dir：____
00AC：有 / 无 / 已读 / 未读
00AG：有 / 无 / 已读 / 未读
00AI：有 / 无 / 已读 / 未读
00AM：有 / 无 / 已读 / 未读
00I_Codex材料池判定：有 / 无 / md+json齐 / 不齐
00L_Codex精读材料词：有 / 无 / md+json齐 / 不齐
00M_Codex写作前原文通读摘抄：有 / 无 / md+json齐 / 不齐
```

只要 package_dir 为空，或四件套缺任一项，就不准查库后直接回答，只能退回入口并报告：

```text
阻断：未进入聚拢库/材料池。
```

四件套已齐但 00I/00L/00M 缺任一项，也不准回答，只能报告：

```text
最终答案硬阻断：聚拢库已进入，材料池到最终写作前闭环未完成。
```

## 二、唯一正路

正式查询只能走：

```text
用户问题
-> 过全库地图
-> 进入聚拢库 / 聚拢总图
-> 抽入口线索
-> 判断问题类型
-> 选择相关库或工具
-> 从相关库取得规范名、标签、入口词、候选编号
-> 全部映射回聚拢总图
-> 在图内放大、缩小、交集、串域
-> 必要时用全文穷尽工具补编号
-> 补出来的编号仍然回聚拢总图
-> 回原文裁判
-> 入材料池
-> Codex 材料池四态判定：可用主证 / 背景材料 / 不可用 / 需补证
-> 精读材料词 / 聚拢裁判单
-> 写作前原文追证摘抄
-> 红楼解语
-> 终稿复核
```

核心铁律：

```text
库不是终点。
检索不是终点。
专题不是终点。
摘要不是终点。
命中不是终点。
召回充分不是终点。
聚拢总图负责盘路。
原文负责裁判。
材料池四态负责能否入写作。
写作前原文追证负责最终证据底稿。
```

## 二点五、状态机总规则

从本入口开始，所有题都分三层状态。

```text
1. 召回状态：
   未命中 / 已命中 / 命中充分 / 命中不足

2. 聚拢裁判状态：
   未裁判 / 裁判中 / 已裁判 / 裁判不足需补证

3. 最终写作状态：
   不可写 / 可写草案 / 可写最终答案
```

硬规则：

```text
无论召回多充分，只要聚拢裁判状态不是“已裁判”，最终写作状态永远不能变成“可写最终答案”。
```

因此旧说法全部降级：

```text
旧错路：可直接命中 -> 可直接写红楼解语
新：召回已命中 -> 待聚拢裁判
```

以后不再按题型设例外。
简单题走极简聚拢裁判。
复杂题走完整聚拢裁判。
区别只在裁判厚薄，不在能不能绕过。

## 三、SQLite 和全文检索的位置

SQLite、FTS、chapters、segments、search_documents 可以用，但不能抢前门。

它们的新位置是：

```text
聚拢总图内部工具
全文穷尽补点工具
原文裁判工具
```

特别判定：

```text
formal_honglou_search.sqlite 是正式底库全文检索原型。
它不是聚拢库本身。
它只能在 00AC/00AG/00AI/00AM 已经形成后，作为聚拢场内部的召回和原文裁判工具使用。
正式路径只能是 outputs/正式底库全文检索原型/formal_honglou_search.sqlite。
工程根目录或 work/ 下的同名文件，不得当作入口或证据库。
```

禁止：

```text
进入红楼梦工程
-> 直接找 sqlite
-> 直接查 chapters.full_text
-> 直接查 segments
-> 直接根据命中写答案
```

允许：

```text
进入红楼梦工程
-> 过 000/128/127/117/119/120/126
-> 进入聚拢总图
-> 确认本题需要全文穷尽补点或原文裁判
-> 再查 sqlite / FTS / chapters / segments
-> 命中点回聚拢总图
-> 原文裁判
```
